针对 P1（DAG-LLM）中“将复杂任务分解为有向无环图（DAG）”这一核心步骤，我们需要设计一个精准的 DAG Decomposer Prompt。

这个 Prompt 的目标是：输入一个自然语言描述的复杂问题，输出一个结构化的、机器可读的 DAG 定义（包含节点和边）。

以下是为您定制的 DAG任务分解Prompt模板，以及针对不同场景的变体。

核心Prompt模板：通用DAG分解器

这个模板适用于大多数通用推理任务（如数学、逻辑、规划）。

你是一个高级任务分解专家。你的任务是将用户提出的复杂问题分解为一个有向无环图（DAG）结构，以便进行并行化推理。

请严格遵循以下步骤：
1.  分析依赖：仔细阅读问题，识别出其中包含的子任务，并分析这些子任务之间的逻辑依赖关系（例如：任务B必须在任务A完成后才能开始，或者任务A和任务B可以同时进行）。
2.  定义节点：将每个独立的子任务定义为一个节点。节点描述应简洁明确，聚焦于具体动作。
3.  定义边：明确表示节点之间的依赖关系。如果节点B的输入依赖于节点A的输出，请建立一条从A到B的有向边。
4.  输出格式：请严格按照以下JSON格式输出，不要包含任何其他解释或文本。

{
  "nodes": [
    {"id": "A", "description": "子任务A的具体描述"},
    {"id": "B", "description": "子任务B的具体描述"},
    {"id": "C", "description": "子任务C的具体描述"}
  ],
  "edges": [
    {"from": "A", "to": "C"},
    {"from": "B", "to": "C"}
  ]
}

现在，请分解以下问题：
[在此处插入用户的问题]

️ 场景化Prompt变体

为了提升特定领域的分解质量，可以在核心模板的基础上进行微调。

1. 数学与逻辑推理（侧重步骤顺序）
适用场景：多步计算、几何证明、应用题。
修改点：强调“已知条件”、“中间变量”和“最终结论”的流动。

Prompt 特征词：
“请识别问题中的已知条件和未知量。将求解过程分解为计算中间变量的步骤，确保依赖关系清晰。”

2. 代码生成（侧重模块化）
适用场景：生成复杂程序、脚本编写。
修改点：强调函数、类、变量作用域和调用关系。

你是一个资深软件架构师。请将用户的需求分解为模块化的代码生成任务DAG。
- 节点类型可以是：定义类、编写函数、实现接口、编写测试用例等。
- 边表示模块间的依赖（例如：主函数依赖于工具函数，测试用例依赖于被测函数）。

输出格式同上（JSON）。

3. 科研与内容创作（侧重并行探索）
适用场景：撰写论文、市场分析报告、假设生成。
修改点：强调并行分支（如不同观点的对比、不同数据的收集）。

Prompt 特征词：
“识别可以并行收集信息的分支（例如：收集A观点的资料和收集B观点的资料）。将任务分解为数据收集、分析、综合和结论四个阶段，明确阶段间的依赖。”

示例演示

为了验证这个 Prompt 的效果，我们可以看一个具体的例子。

用户输入问题
“一个汽车以60km/h的速度行驶了2小时，然后以80km/h的速度行驶了30分钟。如果每100公里消耗5升油，请计算总共消耗了多少油。”

预期的DAG结构
- 节点A：计算第一段路程的距离（速度×时间）。
- 节点B：计算第二段路程的距离（速度×时间）。
- 节点C：计算总距离（A+B）。
- 节点D：根据总距离计算油耗（总距离/100×5）。
- 依赖关系：A→C, B→C, C→D。

模型输出（符合Prompt格式的JSON）
{
  "nodes": [
    {"id": "A", "description": "计算第一段路程的距离：60km/h × 2h"},
    {"id": "B", "description": "计算第二段路程的距离：80km/h × 0.5h"},
    {"id": "C", "description": "计算总距离：A + B"},
    {"id": "D", "description": "计算总油耗：C / 100 × 5"}
  ],
  "edges": [
    {"from": "A", "to": "C"},
    {"from": "B", "to": "C"},
    {"from": "C", "to": "D"}
  ]
}

提升成功率的技巧（P1工程化建议）

在 P1 的系统实现中，不要仅仅依赖一次 Prompt 调用，建议采用 “校验-重试” 机制：

1.  Schema 校验：接收到模型输出后，首先校验 JSON 格式是否正确。
2.  DAG 合法性检查：
    - 检查是否有孤立节点（无依赖且非起点）。
    - 检查是否有环路（使用拓扑排序算法）。如果检测到环路，说明分解失败。
3.  反馈修正 Prompt：
    如果检测到错误，发送修正指令：
    > “检测到错误：图中存在环路/节点X未被使用。请重新分解问题，确保是一个有效的有向无环图（DAG），且所有节点都服务于最终目标。”

通过这种方式，你可以构建一个鲁棒的 DAG-LLM 前端分解器。
