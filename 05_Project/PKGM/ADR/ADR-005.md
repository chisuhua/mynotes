# ADR-005: Agent 触发与执行策略

**上下文**: PRD 决策 D-03 — Agent 触发方式。基于 Karpathy LLM-Wiki 理念设计。

---

## 1. 背景

PKGM 的核心引擎由 LLM Agent 驱动。需要定义 Agent 的触发机制和执行策略：
- 新素材入库时自动触发
- 定期扫描维护知识健康度
- 用户可随时手动触发特定任务
- 基于 Karpathy LLM-Wiki 理念：LLM 直接定稿，异步审核

---

## 2. 决策

### 决策 A：触发机制 — 三层触发

PKGM 支持三种触发方式，由用户选择使用：

| 触发方式 | 触发条件 | 执行时间 | 用途 |
|---------|---------|---------|------|
| **Cron 自动扫描** | 定时触发 | 后台自动 | 日常摄取 + 定期维护 |
| **手动 @ 触发** | 用户在会话中 @ Agent | 立即执行 | 用户主动发起特定任务 |
| **手动命令** | 用户输入 `/pkgm-*` 命令 | 立即执行 | 指定任务类型 |

#### Cron 自动扫描计划

| 任务 | 频率 | 时间 | 说明 |
|------|------|------|------|
| 新素材摄取 | 每天 1 次 | 凌晨 3:00 | 扫描 `00_Raw_Sources/` 新增文件 |
| 健康扫描 | 每周 1 次 | 周日凌晨 4:00 | 冲突检测、断链检查、孤儿页面 |
| 过期检测 | 每周 1 次 | 周日凌晨 4:00 | 与健康扫描合并执行 |

**理由**：Cron 轮询是 OpenClaw 原生支持的方式，无需额外依赖，Phase 1 足够。Phase 2 可调研 Git Hook 或 OpenClaw 原生 watch 能力。

---

### 决策 B：手动触发 — 随时可用

用户可通过以下方式随时触发 Agent：

#### 手动 @ 触发

```
用户: @PKGM 帮我解析 00_Raw_Sources/papers/new-paper.pdf
Agent: 收到，开始解析...

用户: @PKGM 检查一下有没有冲突的知识
Agent: 正在扫描冲突...

用户: @PKGM 帮我生成 CUDA Warp Divergence 的 Wiki 页面
Agent: 正在生成...
```

#### 手动命令（可选，Phase 2 实现）

| 命令 | 作用 |
|------|------|
| `/pkgm-ingest [path]` | 解析指定素材 |
| `/pkgm-scan` | 执行健康扫描 |
| `/pkgm-generate [title]` | 生成指定概念的 Wiki 页面 |
| `/pkgm-status` | 查看 PKGM 当前状态 |

### 决策 C：执行权限 — LLM 直接定稿（Karpathy 理念）

基于 Karpathy LLM-Wiki 的核心理念：**LLM 是知识的直接作者，不是"草稿生成器"**。

#### 工作流

```
新素材放入 00_Raw_Sources/
         │
         ▼ (Cron 或手动触发)
    ┌─────────┐
    │ Ingest  │  解析素材，提取结构化文本
    └────┬────┘
         ▼
    ┌─────────┐
    │ Extract │  实体识别 + 关系抽取 + 溯源标注
    └────┬────┘
         ▼
    ┌─────────┐
    │ Link    │  关联已有知识，建立图谱边
    └────┬────┘
         ▼
    ┌─────────┐
    │ Write   │  直接写入 01_Wiki/（confirmed 状态）
    └────┬────┘
         ▼
    用户异步审核:
      ├── 满意 → 无需操作
      ├── 修改 → 用户直接编辑 Markdown 文件
      └── 推翻 → 用户删除或标记 deprecated
```

#### 溯源保障

Agent 生成的页面通过溯源字段标记 AI 生成状态：

```yaml
---
title: "示例概念"
type: concept

# 溯源
source_type: primary
source_ref: "来源描述"
confidence: 3                     # AI 生成的默认置信度（不是 5 星）

# 验证
verification:
  status: "pending"               # 待用户异步审核

# 创建/更新
created: 2026-04-14
created_by: "agent/pkgm"          # 标记为 AI 生成
updated: 2026-04-14
updated_by: "agent/pkgm"
---
```

#### 用户修改后的处理

| 用户操作 | Agent 响应 |
|---------|-----------|
| 用户修改了内容 | `updated_by` 变为 `"Suhua"`，`confidence` 升级为 5 |
| 用户标记 `deprecated` | Agent 在健康扫描时跳过该页面 |
| 用户删除页面 | Agent 下次生成时不再自动创建 |
| 用户添加新关系 | Agent 学习用户的关联偏好 |

**核心原则**：
- ✅ **不阻塞**：Agent 不需要等用户确认才能写入
- ✅ **可撤销**：用户随时可以修改 Agent 生成的内容
- ✅ **增量学习**：Agent 通过对比用户修改来学习偏好
- ✅ **可追溯**：`created_by` 字段明确标记谁是作者

### 决策 D：单 Agent + Skills 组合

基于 Karpathy 理念：**不是多 Agent 互相调用，而是一个 Agent 根据任务选择不同的工具/技能组合**。

#### Phase 1 可用 Skills 清单（已安装）

| 管线阶段 | Skill | 作用 | 状态 |
|---------|-------|------|------|
| **Ingest** | `obsidian` | Markdown/知识库操作，`[[wikilink]]` 管理 | ✅ 已安装 |
| **Ingest** | `summarize` | 素材摘要 | ✅ 已安装 |
| **Ingest** | `nano-pdf` | PDF 编辑/解析 | ✅ 已安装 |
| **Extract** | `multi-search-engine` | 补充搜索外部信息 | ✅ 已安装 |
| **Extract** | `web-search` | 联网验证/补充 | ✅ 已安装 |
| **Extract** | `agent-browser` | 网页内容抓取 | ✅ 已安装 |
| **Link** | `obsidian` | `[[wikilink]]` 创建和管理 | ✅ 已安装 |
| **Verify** | `self-improving-agent` | 从错误中学习 | ✅ 已安装 |
| **Verify** | `skill-creator` | 需要时创建新 Skill | ✅ 已安装 |

#### 管线执行模式

```
单 Agent PKGM:
  ├── 新素材入库 → 组合 [summarize + obsidian + multi-search-engine]
  ├── PDF 解析   → 组合 [nano-pdf + summarize + obsidian]
  ├── 关系抽取   → 组合 [obsidian + web-search]
  ├── 网页素材   → 组合 [agent-browser + summarize + obsidian]
  ├── 冲突检测   → 组合 [obsidian + self-improving-agent]
  └── 健康扫描   → 组合 [obsidian]
```

#### Phase 2 扩展 Skills（按需安装）

| Skill | 来源 | 作用 |
|-------|------|------|
| `paper-summarizer` | clawhub 安装 | 学术论文深度摘要 |
| `scholar-search` | clawhub 安装 | 学术搜索（arXiv 等） |
| `pdf-extract` | clawhub 安装 | PDF 结构化解析（表格/公式提取） |

## 3. Agent 管线详细设计

### 3.1 Ingest（摄取）

**输入**：`00_Raw_Sources/` 中的新文件 或 用户指定的文件路径

**处理流程**：
1. 检测文件类型（PDF / Markdown / 网页 URL）
2. 解析文件内容
3. 提取元数据（标题、作者、日期等）
4. 生成摘要

**输出**：结构化 JSON + 摘要文本

### 3.2 Extract（抽取）

**输入**：Ingest 输出的结构化内容

**处理流程**：
1. 实体识别（概念、人物、组织、方法等）
2. 关系抽取（依赖、启发、验证、引用等）
3. 溯源标注（来源类型、置信度）

**输出**：实体列表 + 关系三元组

### 3.3 Link（关联）

**输入**：Extract 输出的关系三元组 + 现有 Wiki 页面

**处理流程**：
1. 实体对齐（匹配到已有页面或创建新页面）
2. 更新 frontmatter `relations` 字段
3. 更新正文"关联"区
4. 同步 `[[wikilink]]`

**输出**：更新后的 Wiki 页面

### 3.4 Write（写入）

**输入**：Link 输出的完整页面内容

**处理流程**：
1. 生成 slug 文件名
2. 写入 `01_Wiki/` 对应目录
3. 更新 `01_Wiki/index.md` 索引

**输出**：新的/更新的 Wiki 页面

### 3.5 Verify（验证）

**输入**：全部 Wiki 页面

**处理流程**：
1. 冲突检测（CONTRADICTS 关系）
2. 断链检查（`[[wikilink]]` 目标不存在）
3. 孤儿页面检测（无入链页面）
4. 溯源完整性检查

**输出**：健康扫描报告

## 4. 影响分析

### 正面影响

| 影响 | 说明 |
|------|------|
| 高效流 | LLM 直接定稿，不阻塞用户工作 |
| 灵活触发 | Cron + 手动 @ + 命令，满足各种场景 |
| 可追溯 | 溯源字段明确标记 AI 生成 vs 人工修改 |
| 增量学习 | Agent 从用户修改中学习偏好 |
| 简单架构 | 单 Agent + Skills，Phase 1 易实现 |

### 风险

| 风险 | 缓解措施 |
|------|---------|
| AI 生成内容质量不稳定 | `confidence: 3` 标记 + 异步审核 + 用户可随时修改 |
| Cron 扫描遗漏新文件 | 手动 @ 触发作为补充 |
| Skills 能力不足 | Phase 2 按需安装新 Skills |

---

## 5. 替代方案

| 方案 | 优点 | 缺点 | 未选原因 |
|------|------|------|---------|
| 草稿 → 审核 → 确认 | 质量控制严格 | 阻塞用户工作流 | 违背 Karpathy 高效理念 |
| 多 Agent 协作 | 专业化分工 | 架构复杂，Phase 1 过度设计 | 单 Agent + Skills 更简单 |
| Git Hook 实时触发 | 零延迟 | 配置复杂，跨环境迁移麻烦 | Cron 轮询 Phase 1 足够 |
