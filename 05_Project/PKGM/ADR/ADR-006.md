# ADR-006: 检索策略（分阶段规划）

**状态**: ✅ Confirmed
**确认日期**: 2026-04-14
**确认人**: Suhua
**日期**: 2026-04-14

**上下文**: PRD 决策 D-04 — Phase 1 不做语义检索，先纯图谱关系

---

## 1. 背景

PKGM 需要支持多种检索方式，但需分阶段实施：
- Phase 1：纯图谱关系遍历（关键词 + 图谱遍历 + 元数据过滤）
- Phase 2：语义检索（向量搜索）
- Phase 3：混合检索（语义 + 图谱 + 元数据，LLM 仲裁融合）

---

## 2. 决策

### 决策 A：Phase 1 — 脚本解析 frontmatter 构建内存图

Phase 1 不引入外部数据库，使用 Python 脚本直接解析 Markdown 文件的 YAML frontmatter 构建内存图索引。

#### 架构

```
01_Wiki/*.md 文件
    │
    ▼ (scan.py 脚本)
 解析所有 YAML frontmatter → 构建内存图 (nodes + edges)
    │
    ▼
 支持查询:
   ├── 关键词搜索 (grep title/内容)
   ├── 关系遍历 (DEPENDS_ON, INSPIRED_BY, SURPASSES 等)
   ├── 元数据过滤 (source_type, confidence, verification.status)
   └── 组合查询 (关系 + 元数据)
```

#### 为什么不是纯 grep

| 方案 | 能做什么 | 不能做什么 |
|------|---------|-----------|
| 纯 grep | 匹配 `[[wikilink]]` | 无法获取关系类型、属性、置信度过滤 |
| 解析 frontmatter | 类型化关系、属性过滤 | 需要 Python 环境（可接受） |

#### 查询能力

**1. 关键词搜索**：
```bash
# 搜索标题或内容包含特定关键词的页面
$ python scan.py search "CUDA Warp"
```

**2. 关系遍历**：
```bash
# 查找某个概念依赖了哪些其他概念
$ python scan.py relations "CUDA Warp Divergence" --type depends_on

# 反向查询：哪些概念依赖于我
$ python scan.py relations "CUDA Warp Divergence" --reverse

# 查找某个概念的灵感来源论文
$ python scan.py relations "AOS-Universal" --type inspired_by
```

**3. 元数据过滤**：
```bash
# 查找未验证的知识
$ python scan.py filter --verification unverified

# 查找低置信度的知识
$ python scan.py filter --confidence-lt 3

# 查找特定来源类型的知识
$ python scan.py filter --source-type primary
```

**4. 组合查询**：
```bash
# 查找所有被验证过的、且被至少 2 个其他概念依赖的知识
$ python scan.py query --verified --min-inbound 2
```

#### 内存图数据结构

```python
class KnowledgeGraph:
    nodes: Dict[str, Node]          # slug → Node
    edges: Dict[str, List[Edge]]     # source_slug → [Edge, ...]
    
class Node:
    slug: str
    title: str
    type: str                        # concept, paper, person, ...
    domain: str
    source_type: str
    confidence: int
    verification: dict
    lifecycle: dict
    relations: dict                  # depends_on: [slug, ...], ...
    
class Edge:
    source: str
    target: str
    type: str                        # depends_on, inspired_by, ...
    properties: dict                 # metric, value_delta, result, ...
```

#### 实现约束

- 单次扫描时间 < 5 秒（1000 个文件以内）
- 内存占用 < 50MB
- 支持增量扫描（只解析变更的文件）

### 决策 B：Phase 2 — ChromaDB 向量库

Phase 2 引入向量搜索，选型 **ChromaDB**。

#### 选型对比

| 方案 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| **ChromaDB** | Python 原生，嵌入式，零运维，API 简单，持久化存储 | 大规模（10万+）性能一般 | **✅ 采用** |
| FAISS | 大规模搜索快 | 需要自己管理元数据，安装复杂 | Phase 3 再评估 |
| 无外部依赖（纯向量计算） | 无依赖 | 功能受限，需要自己实现索引 | 不适用 |

#### 使用方式

```python
import chromadb

# 初始化（嵌入式，零运维）
client = chromadb.PersistentClient(path="./03_Engine/chroma_db/")

# 创建集合
collection = client.get_or_create_collection(
    name="pkgm_wiki",
    metadata={"description": "PKGM Wiki pages"}
)

# 索引页面
collection.add(
    documents=[page_content],           # Markdown 正文
    metadatas=[{"slug": "cuda-warp-divergence", "type": "concept"}],
    ids=["cuda-warp-divergence"]
)

# 语义搜索
results = collection.query(
    query_texts=["GPU 缓存优化策略"],
    n_results=5,
    where={"type": "concept"}           # 元数据过滤
)
```

#### 向量来源

- 索引内容 = Markdown 正文（去除 frontmatter 和 `[[wikilink]]` 标记）
- 元数据 = frontmatter 中的字段（type, domain, source_type, confidence, tags）
- 增量更新 = 文件变更时自动更新对应 chunk

#### 安装

```bash
pip install chromadb
```

零运维，嵌入式运行，无需额外服务。

### 决策 C：Phase 3 — LLM 仲裁融合

Phase 3 实现混合检索，采用 **LLM 仲裁融合** 策略。

#### 架构

```
用户查询: "CUDA 12 有哪些新特性？"
    │
    ├── 关键词匹配 (grep) ───────────────→ 结果集 A [3 个页面]
    ├── 图谱遍历 (scan.py) ──────────────→ 结果集 B [5 个相关概念]
    ├── 语义搜索 (ChromaDB) ────────────→ 结果集 C [4 个相似页面]
    └── 元数据过滤 (frontmatter) ────────→ 结果集 D [2 个高置信度页面]
              │
              ▼
         去重 + 排序
              │
              ▼
         LLM 仲裁融合
              │
              ▼
         最终回答 + 引用来源
```

#### LLM 仲裁 Prompt 模板

```
你收到了多路检索结果：

## 关键词匹配结果 (3 个)
1. [[CUDA 12.0 新特性]] - 介绍了 CUDA 12.0 的...
2. [[Hopper 架构]] - Hopper 需要 CUDA 12+...
3. [[nvcc 编译器更新]] - nvcc 12.0 支持...

## 图谱关系结果 (5 个)
- CUDA 12.0 --(OBSOLETES)--> CUDA 11.x
- CUDA 12.0 --(USED_IN)--> PTX-EMU
- CUDA 12.0 --(BELONGS_TO)--> 12_CUDA_Ecosystem

## 语义搜索结果 (4 个)
1. [[GPU Driver API 演进]] - 相似度 0.85
2. [[CUDA 兼容性矩阵]] - 相似度 0.78

## 元数据过滤结果 (2 个)
- [[CUDA 12.0 新特性]] (confidence: 5, verified)
- [[Hopper 架构]] (confidence: 4, verified)

请综合以上结果，生成回答。

要求：
1. 每个结论必须标注来源页面（使用 [[wikilink]] 格式）
2. 优先引用高置信度 (confidence >= 4) 的内容
3. 如果有冲突，指出冲突并标注双方来源
4. 按相关性排序，最相关的内容放在前面
```

#### 融合策略

| 来源 | 权重 | 说明 |
|------|------|------|
| 图谱关系 | 高 | 显式关系，信息量最大 |
| 元数据过滤 | 高 | 置信度/验证状态强信号 |
| 语义搜索 | 中 | 发现隐性关联 |
| 关键词匹配 | 低 | 精确匹配但召回率低 |

LLM 仲裁的优势：
- ✅ 自动处理冲突结果
- ✅ 天然支持溯源引用
- ✅ 可以按用户查询意图动态调整权重
- ✅ 不需要写大量硬编码融合规则

## 3. 检索场景优先级

| 场景 | Phase | 检索方式 | 示例查询 | 实现复杂度 |
|------|-------|---------|---------|-----------|
| 概念定义 | 1 | 关键词匹配 | "什么是 CUDA Warp Divergence？" | ⭐ |
| 关系查询 | 1 | 图谱遍历 | "AOS-Universal 参考了哪些论文？" | ⭐⭐ |
| 未验证知识 | 1 | 元数据过滤 | "哪些知识还没验证过？" | ⭐ |
| 冲突检测 | 1 | 图谱遍历 | "有没有互相矛盾的知识？" | ⭐⭐ |
| 语义搜索 | 2 | 向量搜索 | "帮我找关于 GPU 缓存优化的资料" | ⭐⭐⭐ |
| 混合检索 | 3 | 多路召回 + LLM | "CUDA 12 有哪些新特性？" | ⭐⭐⭐⭐ |

## 4. 影响分析

### 正面影响

| 影响 | 说明 |
|------|------|
| 渐进式实现 | Phase 1 零依赖即可用，后续按需升级 |
| 轻量级 | ChromaDB 嵌入式，无需额外服务 |
| 可溯源 | LLM 仲裁天然支持引用来源 |
| 灵活 | 三种检索方式可按需组合 |

### 风险

| 风险 | 缓解措施 |
|------|---------|
| Phase 1 搜索能力有限 | 手动 @ Agent 补充自然语言查询 |
| ChromaDB 大规模性能 | 超过 10 万页面时再评估 FAISS |
| LLM 仲裁成本高 | 只在混合检索时触发，简单查询直接返回 |

## 5. 替代方案

### 外部工具评估

| 工具 | 评估结果 | 决策 |
|------|---------|------|
| **Graphify** (`safishamsi/graphify`) | 多模态解析、AST 代码分析、交互式图谱可视化 | **Phase 2 补充**，用于批量知识提取和交叉验证，不依赖 |
| Obsidian Graph View | 原生 `[[wikilink]]` 图谱可视化 | **Phase 1 默认方案**，开箱即用 |

## 8. 替代方案

| 方案 | 优点 | 缺点 | 未选原因 |
|------|------|------|---------|
| 纯 grep + 正则 | 最简单 | 不支持关系查询 | 信息量不足 |
| Neo4j 图数据库 | 查询能力强 | 引入外部依赖 | 违反 Phase 1 Markdown 原生原则 |
| Elasticsearch | 全文检索成熟 | 重量级，运维成本高 | 个人知识库杀鸡用牛刀 |
| 规则融合 | 可控 | 需要大量硬编码权重 | LLM 仲裁更灵活 |
