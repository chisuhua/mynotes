## 1. 背景

ADR-001 已确立核心原则（Markdown 文件 = 图谱节点，`[[wikilink]]` = 隐式边，frontmatter relations = 显式边）。
本 ADR 聚焦存储层的具体格式规范，解决 5 个待决策项。

---

## 2. 决策

### 2.1 决策 A：有属性的关系使用扩展 YAML 格式

关系值支持两种格式，Agent 解析时自动兼容：

**简单关系**（无额外属性，字符串列表）：

```yaml
relations:
  depends_on:
    - "[[SIMT Execution Model]]"
    - "[[CUDA Warp]]"
  belongs_to:
    - "[[01_GPU_Architecture]]"
```

**复杂关系**（有属性，对象列表）：

```yaml
relations:
  surpasses:
    - target: "[[Mistral-7B]]"
      metric: "MMLU Score"
      value_delta: "+5.2%"
      dataset: "MMLU-57"
  verified_by:
    - target: "[[PTX-EMU: Warp Divergence 性能测试]]"
      result: "confirmed"
      date: "2026-04-14"
  contradicts:
    - target: "[[CUDA 合并访问策略 v1]]"
      context: "Hopper 架构行为已改变"
```

**解析规则**：
- 字符串值 → 自动提取为无属性关系
- 对象值 → `target` 字段为目标节点，其余为关系属性
- 两种格式可在同一关系类型下混合使用

### 2.2 决策 B：文件使用 slug 命名

文件名基于 `title` 字段生成 slug：
- 小写英文 + 连字符分隔
- 中文标题转为拼音或英文翻译
- 禁止特殊字符（空格、标点、中文）

| title | 文件名 |
|-------|--------|
| `"CUDA Warp Divergence"` | `cuda-warp-divergence.md` |
| `"Attention Is All You Need"` | `attention-is-all-you-need.md` |
| `"PTX-EMU: Warp Divergence 性能测试"` | `ptx-emu-warp-divergence-performance-test.md` |
| `"Ampere GPU Architecture"` | `ampere-gpu-architecture.md` |

**理由**：跨平台兼容、URL 友好、Git diff 可读、Obsidian 无差异。

### 2.3 决策 C：正文"关联"区保留（与 frontmatter 重复）

页面底部必须包含"关联"区，用中文关系名展示：

```markdown
## 关联

- **依赖**: [[SIMT Execution Model]], [[CUDA Warp]]
- **验证**: [[PTX-EMU: Warp Divergence 性能测试]] (confirmed)
- **冲突**: [[CUDA 合并访问策略 v1]] (Hopper 架构行为已改变)
```

**利弊评估**：
- ✅ 人打开页面直接看到关联，无需翻到顶部看 YAML
- ✅ Obsidian Backlinks 面板与手动关联区互补
- ⚠️ 与 frontmatter 内容重复
- **缓解措施**：Agent 生成/更新页面时自动同步两处

### 2.4 决策 D：关系名使用英文（机器），正文使用中文（人读）

| 层级 | 关系名 | 示例 |
|------|--------|------|
| **frontmatter**（机器解析） | 英文 snake_case | `depends_on`, `inspired_by`, `verified_by` |
| **正文关联区**（人读展示） | 中文 | 依赖, 灵感来源, 验证 |

**完整映射表**：

| 英文 key | 中文显示 | ADR-001 关系 ID |
|----------|---------|----------------|
| `depends_on` | 依赖 | R01 |
| `inspired_by` | 灵感来源 | R02 |
| `implements` | 实现 | R03 |
| `verified_by` | 验证 | R04 |
| `obsoletes` | 淘汰 | R05 |
| `refines` | 细化 | R06 |
| `contradicts` | 冲突 | R07 |
| `belongs_to` | 归属 | R08 |
| `cites` | 引用 | R09 |
| `surpasses` | 超越 | R10 |
| `refutes` | 反驳 | R11 |
| `proposes` | 提出 | R12 |
| `created_by` | 创建者 | R13 |
| `affiliated_with` | 所属 | R14 |
| `used_in` | 应用于 | R15 |

### 2.5 决策 E：正文结构自由，最少约束

**必须**：
- YAML frontmatter，包含 `title`、`type`、`source_type` 三个字段

**推荐**（Agent 生成时自动添加）：
- `## 定义` / `## 概述` — 概念解释
- `## 我的理解 🧠` — 原创思考
- `## 来源 📖` — 溯源信息
- `## 关联` — 底部关系展示

**自由**：
- 其他章节、标题、内容完全自由
- 不强制使用特定章节模板
- 允许代码块、Mermaid 图、表格等任意 Markdown 元素

## 4. 隐式边 vs 显式边规则

```
显式边（relations:）覆盖隐式边（正文 [[wikilink]]）

即：
  relations.depends_on 包含 [[B]] → 生成 A --(DEPENDS_ON)--> B
  正文仅有 [[B]] 但 relations 未声明 → 生成 A --(linked_to)--> B
```

**双向链接**由 Agent 动态计算，无需写入目标页面文件。

## 5. 影响分析

### 正面影响

| 影响 | 说明 |
|------|------|
| Obsidian 完全兼容 | `[[wikilink]]` + frontmatter |
| 人读体验好 | 正文关联区用中文，一目了然 |
| Agent 可解析 | YAML frontmatter 标准格式 |
| 跨平台安全 | slug 文件名避免中文路径问题 |
| 灵活度高 | 正文结构自由，不被模板束缚 |

### 风险

| 风险 | 缓解措施 |
|------|---------|
| frontmatter 与正文关联区不同步 | Agent 生成/更新时自动同步 |
| slug 命名丢失中文语义 | `title` 字段保留中文原文 |

---

## 6. 替代方案

| 方案 | 优点 | 缺点 | 未选原因 |
|------|------|------|---------|
| 中文文件名 | 直观 | 跨平台兼容性差 | 违反 Git-based 原则 |
| 纯 frontmatter 无正文关联区 | 无冗余 | 人读体验差 | 人机协同不友好 |
| 固定章节模板 | 一致性好 | 限制自由度 | 不同实体需要不同结构 |
