# ADR-002: 溯源体系（Provenance Schema）

**状态**: ✅ Confirmed
**确认日期**: 2026-04-14
**确认人**: Suhua
**日期**: 2026-04-14
**决策者**: Suhua
**上下文**: ADR-001 已确认，需要定义知识的来源追踪、置信度计算、生命周期管理

---

## 1. 背景

PKGM 管理来自多源头的知识（外部论文、官方文档、原创思考、二手转述）。
如果没有清晰的溯源体系，会导致：

- 无法区分原创结论和他人观点
- 过时/被推翻的知识无法标记
- Agent 自动生成的内容缺乏可信度标注
- 知识冲突时无法追溯原始出处

---

## 2. 决策

### 2.1 核心溯源字段

每个 Wiki 页面的 frontmatter 必须包含以下溯源属性：

```yaml
# 来源分类
source_type: original | primary | secondary | tertiary

# 具体来源指向（至少填一项）
source_ref: "来源描述"
source_url: "https://..."        # 可选，在线资源
source_doi: "10.xxxx/..."        # 可选，学术论文
source_file: "00_Raw_Sources/..." # 可选，本地文件路径

# 置信度
confidence: 1 | 2 | 3 | 4 | 5    # 1-5 星

# 验证状态
verification:
  status: unverified | pending | verified | refuted
  method: "验证方式（可选）"
  verified_by: "验证人"
  verified_date: "2026-04-14"

# 生命周期
lifecycle:
  status: active | superseded | deprecated | refuted
  superseded_by: "[[新页面名]]"   # 可选
  deprecated_reason: "原因"       # 可选
  deprecated_date: "2026-04-14"   # 可选

# 创建/更新
created: 2026-04-14
created_by: "agent/ingest | Suhua | ..."
updated: 2026-04-14
updated_by: "agent/link | Suhua | ..."
```

### 2.2 来源类型定义

| 类型 | 定义 | 默认置信度 | 示例 |
|------|------|-----------|------|
| **original** | Suhua 的原创思考、设计决策、实验结论 | 5 | 04_Knowledge 下的架构文档 |
| **primary** | 一手资料：官方文档、原始论文、专利、源码 | 4 | NVIDIA 编程指南、arXiv 论文 |
| **secondary** | 对一手资料的解读：技术博客、综述文章 | 3 | 知乎技术文章、CSDN 教程 |
| **tertiary** | 二次以上转述：百科摘要、AI 总结 | 2 | Wikipedia、ChatGPT 总结 |

### 2.3 来源类型自动映射规则

Agent 在摄取知识时，按文件路径自动判定 `source_type`：

```yaml
source_type_auto_rules:
  "00_Raw_Sources/papers/*.pdf":     primary
  "00_Raw_Sources/articles/*.md":    secondary
  "00_Raw_Sources/patent/*.md":      primary
  "00_Raw_Sources/code-snippets/*":  primary
  "04_Knowledge/**":                 original
  "AI 提取摘要 (未人工确认)":         tertiary
  "人工确认后的 AI 提取":             primary   # 确认后升级
```

### 2.4 置信度计算规则

**自动计算基础分**：
```
original   → 5
primary    → 4
secondary  → 3
tertiary   → 2
```

**修正因子**：

| 条件 | 修正 | 说明 |
|------|------|------|
| `verification.status == "verified"` | +1（上限5） | 经过实验/交叉验证 |
| `verification.status == "refuted"` | -1 | 已被证伪 |
| 超过 2 年未更新 | -1 | 可能过时 |
| 3+ 个独立来源交叉印证 | +0.5 | 共识度高 |
| 存在未解决的 CONTRADICTS 关系 | -0.5 | 有争议 |

**人工覆盖**：Suhua 可随时手动修改 `confidence` 值，手动值优先级高于自动计算。

### 2.5 验证状态机

```
unverified ──(实验/审查)──→ verified
     │                          │
     │                          │ (发现错误)
     ↓                          ↓
  pending ──(确认错误)────→  refuted
```

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `unverified` | 默认状态，未经检验 | Agent 自动提取后 |
| `pending` | 有待验证，已标记可疑 | 冲突检测发现矛盾 |
| `verified` | 已通过实验或交叉验证 | 人工确认/实验支撑 |
| `refuted` | 已被证伪 | 新证据推翻原有结论 |

### 2.6 生命周期管理

| 状态 | 含义 | 关联关系 |
|------|------|---------|
| `active` | 当前有效 | — |
| `superseded` | 被新版本替代 | 自动建立 `OBSOLETES` 边指向新版本 |
| `deprecated` | 不再推荐使用 | 保留页面但标记警告 |
| `refuted` | 结论被推翻 | 自动建立 `CONTRADICTS` 或 `REFUTES` 边 |

### 2.7 关系级证据引用

对于有属性的关系（如 SURPASSES、VERIFIED_BY），支持携带证据引用：

```yaml
relations:
  surpasses:
    - target: "[[Mistral-7B]]"
      metric: "MMLU Score"
      value_delta: "+5.2%"
      evidence:                     # Phase 2 启用
        quote: "原文引用..."
        page: 12
        table: "Table 3"
        confidence: 0.95

  verified_by:
    - target: "[[PTX-EMU: 性能测试]]"
      result: "confirmed"
      evidence:
        source: "04_Knowledge/.../benchmark.md"
        date: "2026-04-14"
```

### 2.8 知识加工链（Transformation Chain）

完整记录知识从原材料到当前状态的每一步转换：

```yaml
provenance:
  transformation_chain:
    - step: 1
      action: "extracted"
      actor: "agent/ingest"
      timestamp: "2026-04-14T10:30:00+08:00"
      input: "00_Raw_Sources/papers/attention.pdf"
      output: "01_Wiki/concepts/transformer.md (draft)"
    - step: 2
      action: "reviewed"
      actor: "Suhua"
      timestamp: "2026-04-14T14:00:00+08:00"
      changes: "修正了注意力机制的公式描述"
      output: "01_Wiki/concepts/transformer.md (confirmed)"
```

**Phase 1 简化**：仅保留 `created_by` + `updated_by`，完整 `transformation_chain` 在 Phase 2 启用。

## 3. Phase 1 最小可用模板

```yaml
---
title: "示例概念"
type: concept
domain: "01_GPU_Architecture"

# 溯源
source_type: primary
source_ref: "NVIDIA CUDA C Programming Guide, Ch.32"
source_url: "https://docs.nvidia.com/cuda/..."
confidence: 4
tags: [cuda, gpu, performance]
aliases: [别名1, 别名2]

# 验证
verification:
  status: "unverified"

# 生命周期
lifecycle:
  status: "active"

# 创建/更新
created: 2026-04-14
created_by: "agent/ingest"
updated: 2026-04-14
updated_by: "agent/ingest"

# 关系
relations:
  depends_on:
    - "[[相关概念]]"
  belongs_to:
    - "[[01_GPU_Architecture]]"
---
```

## 4. 影响分析

### 正面影响

| 影响 | 说明 |
|------|------|
| 知识可信度透明 | 一眼看出哪些是原创、哪些是引用 |
| 过时知识可追踪 | `lifecycle` 标记淘汰知识 |
| 冲突知识可定位 | `verification.status == "refuted"` 标记 |
| Agent 行为可审计 | `created_by` / `updated_by` 记录操作者 |

### 风险

| 风险 | 缓解措施 |
|------|---------|
| `source_type` 判定错误 | Agent 按路径规则自动判定，人工可覆盖 |
| 置信度自动计算不准确 | 人工可手动覆盖 |
| `transformation_chain` Phase 1 缺失 | Phase 2 补齐，Phase 1 有 created_by 即可追溯 |

---

## 5. 替代方案

| 方案 | 优点 | 缺点 | 未选原因 |
|------|------|------|---------|
| 侧边 JSON 文件记录溯源 | 结构更灵活 | 与 Markdown 分离，维护成本高 | 不符合 Git-based 原则 |
| 仅记录 source_type | 最简单 | 信息量不足，无法追踪生命周期 | 不满足可溯源需求 |
| 引入外部溯源数据库 | 查询能力强 | 过度设计，Phase 1 不需要 | 违反 Markdown 原生原则 |
