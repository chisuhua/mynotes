## 🔬 二、阶段 P0：机制可行性验证（¥15.5，全 24G）

**阶段目标**：验证蓝图 v3.0 的 5 个核心技术机制是否可运行。全部通过方可进入 P1，任一失败则终止项目或回退标准方案。

### **P0-1：分层注意力基础功能验证（含 3 次重复）**

**实验原理**：
验证 `HierarchicalAttention` 层（块内稠密 + 块间稀疏）能否在 MiniMind2 中正确前向/反向传播，不崩溃、不 NaN。通过 3 次重复排除随机初始化导致的偶然成功。

**具体设计**：
- **模型**：MiniMind2-Tiny（4M，`dim=256, n_layers=4, heads=8`）
- **架构修改**：
  - `local_window_size = 512`（当前块细粒度全量注意力）
  - `global_header_size = 64`（其他块仅标题粗粒度注意力）
  - 动态 KV 拼接：`mixed_kv = cat([local_kv, global_header_kv], dim=1)`
- **数据**：合成 Python 代码（0.1B tokens，随机函数片段）
- **训练**：100 steps，lr=5e-4，BS=32，Seq=512
- **评估指标**：
  - 硬性：训练不 NaN、不崩溃、显存峰值 < 6GB（24G 的 25%）
  - 软性：Loss 曲线单调下降（验证收敛性）

**硬件选择论证**：
- **24G**：速度 ~150k tok/s，时间 0.11h，成本 ¥0.17
- **48G**：速度 ~160k tok/s，时间 0.10h，成本 ¥0.60
- **决策**：**选 24G**，成本仅为 48G 的 **28%**，速度差异 <10%

**成本核算**：¥0.17 × 3 次重复 = **¥0.51**

**决策点**：
- 🟢 **通过**：3 次均成功，进入 P0-2
- 🔴 **失败**：≥2 次失败，**终止项目**（核心机制不可行，避免后续 ¥485 浪费）

### **P0-2：DAG 掩码生成开销测试**

**实验原理**：

**具体设计**：

- **模型**：MiniMind2-Small（26M）
- **数据**：Python 代码（0.4B tokens，AST 分块后构建 DAG）
- **测量指标**：
  - 总训练时间对比（B 组 vs A 组）
  - 掩码生成 CPU 时间（`time.perf_counter`）

**硬件选择论证**：
- **24G**：BS=32，速度 ~140k tok/s，0.48h，成本 ¥0.72
- **48G**：BS=64，速度 ~150k tok/s，0.44h，成本 ¥2.64

**成本核算**：¥0.72 × 2 组 = **¥1.44**

**决策点**：
- 🟢 **通过**：B 组速度 > A 组 80%，掩码生成 <5% 总时间，进入 P0-3

### **P0-3：Adapter 切换延迟与精度测试**

**实验原理**：
验证动态 LoRA Adapter 切换是否能在 50ms 内完成，且切换后模型输出符合 Adapter 特性（无权重污染）。这是蓝图"物理单基座 + 逻辑多模型"的基础。

**具体设计**：
- **基座**：MiniMind2-Small（26M，冻结）
- **Adapters**：
  - Adapter-A：代码生成风格（Python 代码训练）
  - Adapter-B：文档注释风格（Markdown 训练）
- **训练**：各 0.25B tokens（轻量级）
- **测试**：
  - 连续切换 100 次（A→B→A...），测量每次切换时间（`torch.cuda.synchronize`）
  - 切换后生成测试：输入相同提示，验证 A 生成代码、B 生成注释
- **指标**：切换延迟 P99 < 50ms，风格一致性 > 90%

**硬件选择论证**：
- **24G**：可同时驻留 2 个 Adapter（每个 ~2MB），切换仅需修改 `forward` 钩子，无数据传输
- **48G**：同样无传输延迟，但单价高 4 倍
- **决策**：**选 24G**，成本 ¥2.5 vs ¥3.0

**成本核算**：训练 ¥2.5 + 测试 ¥0.5 = **¥3.0**

**决策点**：
- 🟢 **通过**：延迟 <50ms，风格正确，进入 P0-4
- 🟡 **优化**：延迟 50-200ms，改为**常驻内存策略**（显存占用 +10%，但无延迟）
- 🔴 **失败**：延迟 >200ms 或权重污染，**放弃动态路由**，改用静态单一模型

### **P0-4：Header KV 压缩精度损失测试**

**实验原理**：
验证"仅保留块标题 KV、丢弃块内 Token KV"的压缩策略，是否导致跨块引用精度断崖下降。这是蓝图"显存恒定"承诺的核心。

**具体设计**：
- **对比组**：
  - **A 组（全量）**：标准 Transformer，保留所有 Token KV
  - **B 组（Header）**：仅保留每块前 64 tokens 的 KV
  - **C 组（摘要）**：用轻量 MLP 压缩块内信息到 64 tokens（增强版）
- **模型**：MiniMind2-Tiny（26M）
- **任务**：跨块引用检测（Block-1 定义 `x=5`，Block-3 询问 `x` 的值）
- **数据**：0.4B tokens（构造的跨块依赖代码）
- **指标**：跨块引用准确率（A 组为基准 100%，B/C 组相对值）

**硬件选择论证**：
- **24G**：BS=32，显存充足，成本 ¥0.66
- **48G**：无必要，成本 ¥2.64
- **决策**：**选 24G**

**成本核算**：¥0.66 × 3 组 = **¥1.98**

**决策点**：
- 🟢 **通过**：B 组精度 > A 组 85%（损失 <15%），进入 P0-5
- 🟡 **优化**：B 组精度 70-85%，采用 **C 组增强压缩**（+¥0.7）
- 🔴 **失败**：B 组精度 <70%，**放弃 32K 目标**，改为 16K 上下文（全量 KV 可容纳）

### **P0-5 (优化)：QK-Normalization 稳定性验证（深模型行为模拟）**

**实验原理**：
对于计划的 **L=12 深模型**，QK-Norm 可防止 Query/Key 点积值过大导致 softmax 饱和。需用 **L=8 模型**（而非 L=4）模拟，因 L=8 的梯度传播行为更接近 L=12。

**具体设计**：
- **模型**：**MiniMind2-Small（26M，L=8）**（关键优化：原 Tiny L=4 不够准确）
- **对比**：
  - A 组：标准 Attention（无 Norm）
  - B 组：QK-Norm（RMSNorm 在 Q/K 投影后，eps=1e-6）
- **观察指标**：
  - 训练 500 steps 内是否出现 Loss Spike（标准组常见，Norm 组应抑制）
  - 梯度范数稳定性（记录 `clip_grad_norm` 触发频率）
- **数据**：0.5B tokens，BS=16（L=8 深模型需减小 BS）

**硬件选择论证**：
- **24G**：L=8, BS=16，显存 ~20GB，安全，速度 ~80k tok/s
- **成本**：0.5h × 1.5 × 2 组 × 2 次重复 = **¥3.0**（优化后）

**决策点**：
- 🟢 **通过**：B 组 Loss 曲线更平滑，Spike 频率 < A 组 50%，进入 P1
- 🔴 **失败**：移除该组件，改用 Gradient Clipping（norm=1.0）

**P0 阶段小计**：¥0.51 + ¥1.44 + ¥3.0 + ¥1.98 + ¥3.0 = **¥9.93**（含优化后调整，实际执行可控制在 ¥15.5 内，含缓冲）

**决策点（黄金决策点 1）**：

### **P1-3：AST 语义分块 vs 滑窗分块（完整版）**

**实验原理**：
验证 AST 分块（按函数/类边界）是否比滑窗分块（固定长度）在代码任务上表现更好，特别是在跨文件调用场景。

**具体设计**：

- **分块策略 B（AST）**：Tree-sitter 解析，按函数/类边界切分，平均长度 2048，保留依赖元数据
- **模型**：MiniMind2-Small（145M），固定使用分层注意力
- **数据**：Python 代码（1B tokens，含跨文件调用关系）
- **评估任务**：

- **指标**：Top-1 准确率

**硬件选择**：24G，BS=32，速度 ~100k tok/s，训练 10h，成本 ¥15/组

**成本核算**：¥15 × 2 组 = **¥30**

**决策点**：
- 🟢 **AST 优**：跨文件任务准确率 > 滑窗 5% → 采用 AST 分块，进入 P1-4
- 🔴 **无差异**：准确率差异 <2% → 采用滑窗分块（实现更简单）

## 🔬 四、阶段 P2：长上下文压力测试（¥100，全 48G）

**阶段目标**：验证 32K 上下文的可行性。24G 在 32K 下 Batch=1 会 OOM 或速度 <20k tok/s（实际成本更高），**强制 48G**。

### **P2-1：32K 显存压力阶梯测试**

**实验原理**：
验证蓝图"显存恒定"承诺。逐步增加上下文长度，测量显存增长曲线，确认非 $O(N^2)$。

**具体设计**：
- **模型**：MiniMind2（145M）+ 分层注意力
- **测试长度阶梯**：2K → 8K → 16K → 24K → 32K（每个点测试 100 steps）
- **测量指标**：
  - 显存峰值（`torch.cuda.max_memory_allocated`）
  - 有效吞吐（tokens/s）
- **对比组**（可选）：标准 Transformer（相同规模，预期 32K OOM）

**硬件选择**：**强制 48G**（24G 32K 时 BS=1，显存 ~22GB 临界，速度 ~15k tok/s，成本爆炸）

**成本核算**：5 个长度点 × 0.5h × ¥6 = **¥15**

**决策点（黄金决策点 2）**：
- 🟢 **通过**：32K 显存 < 16K 的 1.5 倍（亚线性），且 < 40GB → 进入 P2-2
- 🟡 **临界**：32K 显存 40-46GB（可运行但余量小）→ 进入 P2，但 P3 中限制最大长度为 24K
- 🔴 **失败**：32K OOM 或显存线性增长 → **放弃 32K 特性**，降级为 16K 目标（释放 ¥50 到 P3）

### **P2-2：Needle in Haystack（长程依赖召回）**

**实验原理**：
在 32K 上下文中随机插入关键信息（"针"），测试模型在末尾能否召回。这是长上下文能力的金标准。

**具体设计**：
- **模型**：通过 P2-1 的架构（145M）
- **测试构造**：
  - 32K tokens 的代码文档
  - 在 10%、50%、90% 位置插入 `SECRET_KEY = "12345"`
  - 在末尾询问："SECRET_KEY 的值是多少？"
- **样本数**：每个位置 50 次，共 150 次测试
- **指标**：召回准确率（标准 Transformer 在 32K 通常 <50%）

**硬件选择**：48G（推理成本低）

**成本核算**：**¥5**

**决策点**：
- 🟢 **通过**：平均召回率 > 80% → 保留 32K 宣传
- 🟡 **临界**：召回率 60-80% → 降级为"32K 浏览 + 16K 精确理解"
- 🔴 **失败**：<60% → 架构无法支持长程依赖，需重新设计 KV 压缩策略

### **P2-5 (优化)：课程学习验证（3 阶段渐进）**

**特别关键理由**：分层注意力架构需从短上下文（块内稠密注意力）逐渐适应长上下文（块间稀疏注意力）。若直接训练 2K，模型可能无法学习块间路由；课程学习让模型先掌握局部模式，再学习全局稀疏关联，**避免梯度不匹配**。

**具体设计**：
