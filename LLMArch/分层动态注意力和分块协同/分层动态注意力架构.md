### 🏗️ 一、完整架构配置 (Architecture Specification)

本架构命名为 **MiniMind-MoE-Deep-2.1B**。核心策略是“深度优先”，通过增加层数提升逻辑推理能力，同时控制专家数以优化显存和速度。

#### 1. 全局参数
| 参数项 | 设定值 | 说明 |
| :--- | :--- | :--- |
| **总层数 ($L$)** | **12** | 比标准 MiniMind ($L=8$) 增加 50%，显著提升推理深度。 |
| **隐藏层维度 ($d_{model}$)** | **1152** | 保持较宽的单层表征能力，适配 1152 为 64 的倍数，利于 Tensor Core 加速。 |
| **注意力头数 ($n_{heads}$)** | **18** | $1152 / 18 = 64$ (每头维度)，符合高效 Attention 实现。 |
| **KV 头数 ($n_{kv\_heads}$)** | **6** | GQA (Grouped Query Attention) 比例 3:1，减少 KV Cache 显存，加速推理。 |
| **词汇表大小 ($Vocab$)** | **64,000** | 兼容 Qwen/Baichuan 等主流中文分词器。 |
| **激活函数** | **SwiGLU** | 现代 LLM 标准配置，优于 GeLU。 |
| **位置编码** | **RoPE** | 旋转位置编码，基频 $\theta=10000$。 |
| **归一化** | **RMSNorm** | 前置归一化 (Pre-Norm)，更稳定。 |

#### 2. MoE 专用参数
| 参数项 | 设定值 | 说明 |
| :--- | :--- | :--- |
| **专家总数 ($E$)** | **16** | 相比 $E=32$ 减半，降低路由开销和显存碎片。 |
| **激活专家数 ($k$)** | **2** (Top-2) | 每个 Token 仅由 2 个专家处理，保证稀疏性。 |
| **专家中间维度 ($d_{ff}$)** | **$4 \times d_{model}$** | 即 $4608$。每个专家包含两个线性层 (Up/Down)。 |
| **共享专家 (Shared)** | **0** | 纯 MoE 结构，或可设 1 个共享专家增强稳定性（本方案暂设为 0 以简化计算）。 |
| **路由算法** | **Softmax Top-K** | 带负载平衡损失 (Load Balancing Loss, $\alpha=0.01$)。 |

#### 3. 参数量详细拆解
*   **Embedding 层**: $64,000 \times 1152 \approx$ **73.7 M**
*   **每层稠密部分 (Attention + Norm)**:
    *   QKV Proj: $1152 \times (1152 + 64 \times 2 \times 3) \approx$ **1.8 M** (GQA 优化后)
    *   O Proj: $1152 \times 1152 \approx$ **1.3 M**
    *   Norms: 忽略不计
    *   **小计**: ~3.5 M / 层 $\times$ 12 层 = **42 M**
*   **每层稀疏部分 (MoE MLP)**:
    *   单专家参数: $2 \times (1152 \times 4608) \approx$ **10.6 M**
    *   总专家参数: $16 \times 10.6 \text{ M} \approx$ **170 M** / 层
    *   **12 层总计**: $170 \text{ M} \times 12 \approx$ **2.04 B**
*   **最终输出层 (LM Head)**: $1152 \times 64,000 \approx$ **73.7 M** (若权重绑定则减半)

> **📊 总参数量统计**:
> *   **总参数 ($P_{total}$)**: $73.7 + 42 + 2040 + 73.7 \approx$ **2.23 Billion (2.23B)**
> *   **激活参数 ($P_{active}$)**: Embed + Attn + (2 Experts) + Head
>     *   $73.7 + 42/12 + (2 \times 10.6) + 73.7 \approx$ **0.19 B (每层激活)**
>     *   整体等效激活参数量约为 **0.75 Billion (0.75B)** (考虑所有层累加效应)。

### 💾 二、显存占用详细计算 (Memory Breakdown)

**前提条件**: BF16 精度，Micro-Batch Size (**BS**) = **24**, Seq Len = **2048**, Gradient Checkpointing (GC) 开启。

#### 1. 静态显存 (Static Memory)
*模型权重 + 梯度 + 优化器状态 (AdamW)*

| 组件 | 参数量 (B) | 精度/状态 | 字节/参数 | 计算公式 | 显存占用 (GB) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **模型权重** | 2.23 | BF16 | 2 B | $2.23 \times 2$ | **4.46** |
| **梯度** | 2.23 | BF16 | 2 B | $2.23 \times 2$ | **4.46** |
| **优化器状态** | 2.23 | FP32 (m+v+master) | 12 B | $2.23 \times 12$ | **26.76** |
| **MoE 稀疏优化** | - | - | - | 仅激活专家需梯度/优化器系数修正：$0.2(\text{Dense}) \times 14 + 0.8(\text{Sparse}) \times (2+2+12\times\frac{2}{16})$平均系数 ≈ **8.5 B/param** | **18.96** |
| **静态总计** | | | | **(修正后)** | **~19.0 GB** |

> *注：MoE 的梯度和优化器仅针对激活的 2/16 专家计算，大幅节省显存。*

#### 2. 动态显存 (Dynamic Memory)
*激活值 (Activations) + 临时缓冲区*
公式：$M_{act} \approx C \times L \times d_{model} \times BS \times Seq$
其中 $C \approx 0.00003$ (含 GC 重计算缓冲因子)。

| 组件 | 计算细节 | 显存占用 (GB) |
| :--- | :--- | :--- |
| **Attention 激活** | $12 \text{ (L)} \times 1152 \text{ (d)} \times 24 \text{ (BS)} \times 2048 \text{ (Seq)} \times 2 \text{ (Bytes)} \times \text{GC Factor}(1.5)$ | **12.4** |
| **MoE 激活** | $12 \times 24 \times 2048 \times (\text{Router} + \text{Expert Input/Out})$ | **4.2** |
| **临时缓冲区** | Flash Attention 及矩阵乘法 Workspace | **1.5** |
| **动态总计** | | **~18.1 GB** |

#### 3. 系统与其他 (Overhead)
| 组件 | 说明 | 显存占用 (GB) |
| :--- | :--- | :--- |
| **CUDA 上下文** | 驱动、内核加载 | 1.5 |
| **Python/框架开销** | PyTorch 缓存、数据加载器 | 1.0 |
| **安全余量** | 防止碎片化 OOM | 1.5 |
| **总计** | | **4.0 GB** |

#### 📊 显存总账
$$ M_{total} = 19.0 \text{ (Static)} + 18.1 \text{ (Dynamic)} + 4.0 \text{ (Overhead)} = \mathbf{41.1 \text{ GB}} $$
*   **可用显存**: 48 GB
*   **剩余空间**: **6.9 GB** (非常安全，甚至可尝试 BS=28 或 Seq=2560)

### ⚙️ 三、训练配置 (Training Configuration)

#### 1. 超参数设置
*   **优化器**: AdamW ($\beta_1=0.9, \beta_2=0.95, \epsilon=1e-8$)
*   **学习率**: 峰值 $5 \times 10^{-4}$，Cosine Decay，Warmup 5% (250M tokens)
*   **权重衰减**: 0.1
*   **Gradient Clip**: 1.0
*   **Loss**: CrossEntropy + Load Balancing Loss ($\alpha=0.01$)

#### 2. 并行与加速策略
*   **精度**: **BF16** (原生支持，无需 Loss Scaling)
*   **Flash Attention 2**: **开启** (加速 Attention，降低显存)
*   **Gradient Checkpointing**: **开启** (针对 MoE 层和 Attention 层，牺牲 20% 时间换 50% 显存)
*   **Data Parallel**: DDP (单卡即为 1)
*   **ZeRO**: Stage 1 (仅分片优化器状态，单卡下效果有限但兼容性好)
*   **Compile**: `torch.compile(mode="reduce-overhead")` (可选，预计提速 10-15%)

#### 3. 数据配置
*   **数据集**: `pretrain_hq.jsonl` (中文高质量语料)
*   **Packaging**: Sequence Packing (将短样本拼接至 2048，提升效率)

### 💡 六、风险预案与优化建议

1.  **OOM 风险**:
    *   若干燥运行显存 > 46GB，立即将 **BS 从 24 降至 20**。
    *   代价：时间增加约 20% (至 14h)，成本增至 ¥84，仍在预算内。
2.  **专家负载不均**:
    *   若发现某些专家从未被激活 (Dead Experts)，增加 `load_balancing_loss` 系数至 0.05。
3.  **速度不及预期**:
    *   启用 `torch.compile`。
    *   检查数据加载是否阻塞 (增加 `num_workers` 至 8-16)。
4.  **长文本需求**:
    *   由于显存有 ~7GB 余量，若需增强长文能力，可将 **Seq Len 提升至 2560**，BS 保持 24，显存仍安全。

### 🧠 一、为什么蒸馏能“省数据”？

#### 1. 核心原理：从“自学”变成“名师辅导”
*   **预训练 (Pre-training)**: 模型像小学生一样，从杂乱的网络文本中自己摸索规律。效率低，需要海量数据（Chinchilla 定律：20x 参数）。
*   **蒸馏 (Distillation)**: 模型直接学习“老师”（如 Qwen-72B, GPT-4）已经提炼好的**思维过程**和**高质量答案**。
    *   **信息密度极高**: 1 条蒸馏数据包含的逻辑量 ≈ 100 条普通网页文本。
    *   **收敛极快**: 模型不需要试错，直接模仿最优解。

#### 2. 量化效果
根据近期研究（如 *TinyLlama*, *Qwen-Distill*, *DeepSeek-Distill* 系列）：
*   **1B Tokens 的高质量蒸馏数据** $\approx$ **10B-20B Tokens 的普通预训练数据**。
*   这意味着：如果你用纯蒸馏数据训练，**5B Tokens 可能足以让 2.2B 模型达到“高中生”水平**，而不是“文盲”。

### 🛠️ 二、针对你方案的蒸馏实施策略

既然你的预算锁定在 **5B Tokens (~¥70)**，我们必须把这 5B 用在刀刃上。

#### 方案 A：纯蒸馏模式 (Pure Distillation) —— **强烈推荐**
*   **数据来源**: 完全使用由强模型（Teacher）生成的数据。
    *   **推荐 Teacher**: Qwen-2.5-72B-Instruct, DeepSeek-V3, 或 GPT-4o。
    *   **数据内容**: 数学推理、代码生成、逻辑问答、多步指令遵循。
    *   **格式**: `User Query` -> `Teacher Thought Process (CoT)` -> `Teacher Answer`。
*   **训练目标**: 让学生模型同时学习 **答案** 和 **思维链 (Chain of Thought)**。
*   **预期效果**:
    *   5B Tokens 纯蒸馏 $\rightarrow$ 逻辑能力媲美 20B+ 预训练模型。
    *   特别适合发挥你 **$L=12$** 的深度优势（因为蒸馏数据多为多步推理，正好需要深度网络来处理）。
*   **成本**: 依然是 **¥70** (数据生成是一次性成本，可找开源数据集如 `OpenHermes`, `Cosmopedia`, `Magpie-Pro`)。

#### 方案 B：混合模式 (Hybrid) —— **稳健选择**
*   **数据配比**:
    *   **20% (1B Tokens)**: 高质量通用语料 (维基百科、书籍)。$\rightarrow$ 保证语言流畅度、常识。
    *   **80% (4B Tokens)**: 蒸馏数据 (CoT、代码、理科题)。$\rightarrow$ 注入逻辑和推理能力。
*   **优势**: 避免模型变成“只会做题的机器”，保留一定的语言泛化能力。

### ⚠️ 五、潜在风险与注意事项

1.  **“老师”的偏见**: 学生模型会完美继承老师的幻觉和偏见。务必选择强大的、经过对齐的 Teacher (如 Qwen-2.5-72B-Instruct)。

3.  **分布偏移**: 如果蒸馏数据全是数学题，模型可能不会写诗。
    *   *对策*: 确保蒸馏数据集涵盖 **Code, Math, Reasoning, Creative Writing, General Chat** 五大领域。
