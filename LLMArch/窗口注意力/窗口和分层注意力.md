## 一、摘要 (Executive Summary)

### 1.1 核心架构定位
Step 3.5 Flash 代表了**"系统-模型协同设计"**的工程化巅峰，通过三项关键技术实现 196B 总参数 / 11B 激活参数下的前沿性能：
- **细粒度 MoE**: 288 专家 / Top-8 路由，稀疏度 18:1
- **3:1 混合注意力**: 3层 Sliding Window Attention (SWA, w=512) + 1层 Full Attention
- **MTP-3**: 多词元并行预测，实现 2-3 倍推理加速

### 1.2 关键权衡 (Trade-offs)
| 维度 | 选择 | 放弃 |  rationale |
|------|------|------|------------|
| **注意力模式** | Hybrid SWA (3:1) | Pure Linear Attention | 硬件效率 > 理论复杂度 |
| **窗口策略** | 固定 w=512, d=1 | Dilated/可变窗口 | 密集 GEMM 优化 > 感受野灵活性 |
| **全局机制** | 周期性 Full Attention 层 | 显式 Global Tokens | 通用性 > 任务特定优化 |
| **解码策略** | MTP-3 并行验证 | 纯自回归 | 吞吐率 > 纯自回归质量 |

## 二、基础架构深度解析

### 2.1 MoE 架构设计空间

#### 2.1.1 路由机制对比
- **Switch Routing** (Top-1): 极简但负载不均，专家崩溃风险高
- **Step 3.5 Flash 方案** (Top-8 of 288): 
  - 负载均衡: EP 级显式平衡损失 + 激活裁剪
  - 噪声注入: 训练时向路由 logits 添加高斯噪声，促进专家探索
  - 容量因子: 隐含通过激活裁剪控制，避免少数专家过载

#### 2.1.2 专家粒度
- **细粒度专家** (288 vs DeepSeek 的 256): 更高的专业化程度，适合 Agent 工具调用等离散任务
- **共享专家** (1个始终激活): 捕获通用知识，防止路由崩溃时的信息丢失

### 2.2 混合注意力机制 (Hybrid Attention)

#### 2.2.1 3:1 比例的工程逻辑
```
Layer Pattern: [SWA] [SWA] [SWA] [FULL] [SWA] [SWA] [SWA] [FULL] ...
```
- **感受野计算**: 3层 SWA 堆叠提供 3×512=1536 的局部感受野，FULL 层提供全局同步
- **计算开销**: 相比纯 Full Attention，降低约 75% 计算量；相比纯 SWA，保留关键全局依赖
- **硬件亲和性**: 固定模式允许预编译优化，避免动态稀疏的开销

#### 2.2.2 与替代方案的对比
| 方案 | 复杂度 | 硬件效率 | 长程依赖 | 适用场景 |
|------|--------|----------|----------|----------|
| **Pure Full Attention** | O(n²) | ★ | ★★★ | 短序列、高精度需求 |
| **Step 3.5 Flash (3:1)** | O(n×w) | ★★★ | ★★☆ | 通用 Agent、代码生成 |
| **Dilated SWA (d=4)** | O(n×w) | ★☆ | ★★★ | 超长文档、需要极大感受野 |
| **Linear Attention** | O(n) | ★★ | ★☆ | 极端长序列、流式场景 |
| **Deformable Attention** | O(n×k) | ★ | ★★★ | 视觉任务、动态内容 |

### 2.3 MTP-3 (Multi-Token Prediction-3)

#### 2.3.1 机制细节
- **架构**: 共享 44层 Backbone + 4个独立 MTP Heads
- **训练目标**: 组合损失 L = λ₁L₁ + λ₂L₂ + λ₃L₃ + λ₄L₄（权重递减）
- **推理加速**: 草稿 token 接受率 85-90%，实现 1.8-2.0x 速度提升

#### 2.3.2 与投机解码的关系
MTP-3 是**自投机解码**（Self-Speculative Decoding）：
- 无需 draft model，使用自身 head 生成候选
- 避免分布不匹配问题（传统投机解码的 draft 与 target 分布差异）

## 三、可变窗口与多尺度机制

### 3.1 扩张率 (Dilation Rate) 设计空间

#### 3.1.1 数学定义
- **标准 SWA**: 关注区间 [i-w, i+w]，所有位置
- **Dilated SWA**: 关注位置 {i ± k×d | k=0,1,...,w/d}，间隔为 d
- **有效感受野**: L × d × w（L为层数）

#### 3.1.2 分层扩张策略 (Hierarchical Dilation)
参考 Longformer 的最优实践：
- **底层 (1-6层)**: d=1, w=64-128（局部语法）
- **中层 (7-12层)**: d=2, w=256（句级依赖）
- **顶层 (13+层)**: d=3-4, w=512（文档级关系）

#### 3.1.3 工程约束
- **内存墙**: Dilated attention 的不规则访存模式导致 GPU 利用率下降 30-50%
- **实现方案**: 需 TVM 自定义 CUDA kernel，无法使用标准 FlashAttention

### 3.2 多尺度并行 (Multi-Scale Heads)
在同一层内并行使用不同窗口配置：
- **头 1-4**: w=64, d=1（细粒度）
- **头 5-8**: w=256, d=2（中程）
- **头 9-12**: Full Attention（全局，仅少量头）

**权衡**: 增加表达能力，但破坏内存连续性，实际 wall-clock 时间可能不如均匀 3:1 混合。

## 四、Global-Local 混合注意力进阶

### 4.1 全局 Token 选择策略

#### 4.1.1 静态配置
- **特殊 Token**: [CLS], [QUESTION], [SUMMARY]
- **位置锚点**: 序列前4个 token（StreamingLLM 的 Attention Sink）
- **任务特定**: QA 中的问题词，代码中的函数定义

#### 4.1.2 动态识别
- **Heavy Hitter**: 基于注意力权重动态选择（HyperAttention）
- **Uncertainty-based**: 识别"Wait"、"Let me reconsider"等反思节点（Thought Anchors）
- **压缩感知**: 使用 LLMLingua-2 等轻量级模型评估 token 重要性

### 4.2 CoT-Aware 全局化 (前沿方向)
**核心洞察**: 长链推理中，规划句和不确定性管理句是隐式全局锚点。

**架构设想**:
```
CoT 序列: [Planning] → [Step 1] → [Step 2] → [Reflection] → [Answer]

Attention 掩码:
- Planning/Reflection: 全局（可 attend 全序列，可被全序列 attend）
- Step 1/2: 局部窗口（仅双向邻近）
- Answer: 可 attend 所有前序（标准因果），但仅 Planning/Reflection 可反向 attend 它
```

**实现挑战**: 动态稀疏掩码与 FlashAttention 的兼容性。

## 五、层次化生成与压缩解码 (Hierarchical Decoding)

### 5.1 架构范式
**两阶段生成**:
1. **压缩阶段**: 生成高信息密度 token（短词、句法骨架、语义锚点）
2. **扩展阶段**: 基于压缩表示，通过并行/非自回归机制快速解码为完整文本

### 5.2 技术实现路径

#### 5.2.1 ChainCoder 模式 (代码专用)
- **S1 (结构)**: 类/函数定义 → **S2 (逻辑)**: 算法大纲 → **S3 (实现)**: 具体代码 → **S4 (细节)**: 变量名/格式
- **压缩率**: S1/S2 仅占最终 token 的 20-30%
- **解码策略**: S1/S2 自回归，S3/S4 可并行填充

#### 5.2.2 EAGLE-3 模式 (通用)
- **Draft Head**: 基于 target model 隐藏状态生成特征级草稿 token
- **Tree Attention**: 一次验证 64 个候选 token 的树状结构
- **接受率**: 80%+，延迟降低 2.5x

#### 5.2.3 Gumiho 模式 (计算分配)
- **早期层 (压缩)**: 完整 Transformer，串行生成关键规划 token
- **后期层 (扩展)**: 轻量级 MLP，并行生成剩余 token
- **优势**: 早期高质量保证，后期高吞吐

### 5.3 与 Step 3.5 Flash 的融合点
**MTP-3 扩展为层次化 MTP**:
- **Head 1**: 生成语义压缩 token（规划级）
- **Head 2-4**: 基于压缩 token 并行扩展细节
- **验证**: 主模型一次性验证压缩 token 的正确性，再接受扩展

## 六、实验设计建议

### 6.1 基准测试集

#### 6.1.1 注意力机制评估
- **长程依赖**: Long Range Arena (LRA), SCROLLS
- **代码建模**: SWE-bench Verified, LiveCodeBench (测 256K 上下文代码库理解)
- **Agent 能力**: τ²-Bench (工具调用), BrowseComp (浏览器导航)

#### 6.1.2 效率评估
- **Throughput**: Tokens/s (单流/多流), TPS (Transactions Per Second)
- **内存**: KV Cache 占用 (GB), 峰值显存
- **延迟**: Time-to-First-Token (TTFT), 端到端延迟

### 6.2 消融实验设计

#### 实验 1: 混合比例优化
```
变量: Full Attention 间隔 (1:1, 2:1, 3:1, 4:1, 6:1)
控制: 总层数 45, 窗口大小 512
指标: 长程依赖任务准确率 vs 推理速度
预期: 3:1 或 4:1 为帕累托最优
```

#### 实验 2: 窗口大小与扩张率
```
变量: 
  - 固定窗口 (512, 1024, 2048)
  - 分层窗口 (64→128→256)
  - 扩张率 (d=1,2,4)
控制: 计算量恒定 (FLOPs 匹配)
指标: 感受野有效覆盖, 实际 wall-clock 时间
```

#### 实验 3: Global Token 策略
```
变量:
  - 无全局 (纯 SWA)
  - 固定全局 (前4 token)
  - 任务特定全局 (问题词/锚点)
  - 动态全局 (Heavy Hitter)
指标: 下游任务准确率, 训练稳定性
```

#### 实验 4: 层次化解码
```
阶段 1: 训练压缩 tokenizer (语义级 BPE)
阶段 2: 对比
  - 基线: 纯自回归
  - MTP-3: 并行 4 token
  - 层次化: 先规划 token, 后并行扩展
指标: 生成质量 (BLEU/Pass@1), Tokens/s, 步骤数压缩率
```

### 6.3 训练稳定性技巧
- **MIS-PO**: 针对长链推理的掩码独立采样策略优化，防止梯度爆炸
- **负载均衡**: MoE 的 EP 级平衡损失权重需随训练动态调整
- **Warmup 策略**: 前 10% 训练步使用 Full Attention，逐步引入稀疏模式

## 七、架构创新路线图

### 7.1 短期优化 (3-6个月)
1. **动态 3:1 比例**: 根据输入复杂度动态调整 Full Attention 频率（简单输入 6:1，复杂 2:1）
2. **CoT Sink Token**: 在预训练中加入可学习的 [REASON_START], [REASON_END] token，强制作为全局锚点
3. **MTP-4/8**: 扩展多 token 预测至 8 个，配合更激进的推测解码

### 7.2 中期创新 (6-12个月)
1. **可变形 SWA**: 数据依赖的动态窗口大小（类似 Deformable Conv），关键区域窗口自动扩大
2. **层次化 Hybrid Attention**: 
   - 底层: Linear Attention (O(n)) 处理长上下文预处理
   - 中层: SWA 处理局部精细化
   - 顶层: Full Attention 处理全局决策
3. **语义投机解码**: EAGLE-3 的泛化版本，-draft model 生成语义压缩 token，target model 扩展

### 7.3 长期愿景 (1年+)
1. **Fully Hierarchical LLM**: 
   - L0: 思维锚点 token (Thought Anchors) - 全局可见
   - L1: 推理步骤 token - 局部窗口
   - L2: 表面形式 token - 非自回归并行生成
2. **Adaptive Computation Time**: 根据 token 重要性动态分配计算深度（重要 token 走更多层，简单 token 跳过）
3. **System-Model Co-design**: 与硬件厂商合作，为 Dilated SWA / 层次化解码设计专用稀疏计算单元
