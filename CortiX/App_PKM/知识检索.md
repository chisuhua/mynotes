
## 文档知识提取与整理

面对“大量文档”（如几百份 PDF、Word、技术手册），核心挑战在于：**如何高效 ingestion（摄入）、如何精准检索（RAG）、以及如何将提取的知识结构化输出**。

本方案将 OpenClaw 配置为一个**“企业级知识工厂”**：
1.  **输入**：海量杂乱文档。
2.  **处理**：自动切片、向量化、元数据提取。
3.  **输出**：结构化的知识库（Markdown/JSON）或 基于知识的精准问答。

---

### 第一阶段：环境准备与架构规划

#### 1. 硬件与依赖检查
*   **内存**：建议 16GB+（处理大量文档向量化时消耗较大）。
*   **存储**：确保 `~/.openclaw` 所在分区有足够空间（向量索引 + 原始文件）。
*   **模型**：推荐使用支持长上下文的模型（如 `gemini-3.1-pro-preview` 或 `claude-3.5-sonnet`）用于总结，嵌入模型使用本地轻量级模型（如 `nomic-embed-text` 通过 Ollama）以节省成本和保护隐私。

#### 2. 目录结构规划 (关键)
不要把所有文件扔进一个文件夹。采用**“领域 - 子领域”**的层级结构，便于 OpenClaw 进行隔离索引和权限管理。

```bash
mkdir -p ~/openclaw-knowledge-base
cd ~/openclaw-knowledge-base

# 示例结构
mkdir -p 01_技术架构/后端规范
mkdir -p 01_技术架构/前端规范
mkdir -p 02_产品文档/需求规格书
mkdir -p 03_合规法律/合同模板
mkdir -p 04_历史项目/复盘报告

# 将你的几百个文档按类别放入对应文件夹
# cp /downloads/*.pdf 01_技术架构/后端规范/
```

### 第二阶段：OpenClaw 核心配置 (`openclaw.json`)

我们需要配置两个核心部分：**全局记忆引擎 (Memory Core)** 和 **专用提取 Agent**。

编辑 `~/.openclaw/openclaw.json`：

```json
{
  "models": {
    "providers": {
      "google": {
        "apiKey": "YOUR_GOOGLE_API_KEY",
        "api": "google-generative-ai"
      },
      "ollama": {
        "baseUrl": "http://localhost:11434",
        "api": "openai-completions" 
      }
    },
    "defaults": {
      "primary": "google/gemini-3.1-pro-preview",
      "embedding": "ollama/nomic-embed-text" 
    }
  },
  "agents": {
    "definitions": {
      "knowledge-ingestor": {
        "description": "专门用于批量读取文档并生成结构化摘要的 Agent",
        "model": "google/gemini-3.1-pro-preview",
        "systemPrompt": "你是一个专业的知识工程师。你的任务是读取用户上传的文档内容，提取核心知识点，并按以下 Markdown 格式输出：\n# [文档标题]\n## 核心主题\n## 关键实体 (人/事/物)\n## 重要规则/流程\n## 摘要总结 (200 字以内)\n注意：必须保留原文的关键数据，严禁幻觉。",
        "tools": [
          {
            "name": "fs-read",
            "config": { "allowedPaths": ["~/openclaw-knowledge-base"] }
          }
        ]
      },
      "domain-expert": {
        "description": "基于特定领域知识库的回答专家",
        "model": "google/gemini-3.1-pro-preview",
        "systemPrompt": "你是一名领域专家。请严格基于挂载的知识库片段回答问题。如果知识库中没有答案，请直接说'知识库中未找到相关信息'，不要编造。回答时必须注明引用来源（文件名）。",
        "tools": [
          {
            "name": "memory-core",
            "config": {
              "enabled": true,
              "indexing": {
                "autoIndex": true,
                "watchPaths": [
                  "~/openclaw-knowledge-base/01_技术架构",
                  "~/openclaw-knowledge-base/02_产品文档",
                  "~/openclaw-knowledge-base/03_合规法律"
                ],
                "fileExtensions": [".pdf", ".docx", ".md", ".txt"],
                "chunkSize": 1024, 
                "chunkOverlap": 200
              },
              "retrieval": {
                "topK": 8,
                "threshold": 0.65,
                "includeSources": true,
                "hybridSearch": true 
              }
            }
          }
        ]
      }
    }
  }
}
```

**配置亮点解析**：
*   **双模型策略**：推理用云端强模型 (Gemini)，嵌入用本地模型 (Ollama)，平衡成本与隐私。
*   **Hybrid Search**：开启混合搜索（关键词 + 向量），对于技术文档中的具体错误码、版本号检索至关重要。
*   **WatchPaths**：监控多个目录，实现分领域隔离。
*   **Chunk 设置**：`1024/200` 是通用黄金比例，适合大多数文档。

### 第三阶段：批量摄入与索引 (Ingestion)

配置完成后，执行初始化索引。这是最关键的一步，将非结构化文档转化为向量数据库。

#### 1. 执行全量索引
```bash
# 触发所有 watchPaths 的索引重建
openclaw memory index --all --force
```
*观察终端输出，确保没有报错，且显示 "Indexed XXX chunks from XXX files"。*

#### 2. 验证索引质量 (可选但推荐)
使用 `knowledge-ingestor` Agent 测试单个文件的提取效果：
```bash
# 测试提取某一份复杂 PDF 的核心内容
openclaw chat --agent knowledge-ingestor "请读取 ~/openclaw-knowledge-base/01_技术架构/后端规范/微服务设计指南.pdf 并生成结构化摘要"
```
*检查输出是否符合预期的 Markdown 格式。如果格式不对，调整 Agent 的 `systemPrompt`。*

### 第四阶段：实战工作流

现在你拥有了一个活的知识库，可以进行两种核心操作：

#### 场景 A：知识整理与归档 (批量提取)
如果你想把几百份 PDF 变成几百份结构化的 Markdown 笔记，存入 Notion 或 Obsidian。

**操作步骤**：
1.  编写一个简单的 Shell 脚本 `batch-summarize.sh`：
    ```bash
    #!/bin/bash
    OUTPUT_DIR="./structured-output"
    mkdir -p $OUTPUT_DIR

    # 遍历所有 PDF (示例)
    find ~/openclaw-knowledge-base -name "*.pdf" | while read file; do
      echo "正在处理：$file"
      
      # 调用 OpenClaw 提取
      SUMMARY=$(openclaw chat --agent knowledge-ingestor "请读取 '$file' 并生成结构化摘要，只输出 Markdown 内容，不要废话")
      
      # 保存为同名 md 文件
      FILENAME=$(basename "$file" .pdf)
      echo "$SUMMARY" > "$OUTPUT_DIR/$FILENAME.md"
      
      echo "✅ 已保存至 $OUTPUT_DIR/$FILENAME.md"
    done
    ```
2.  运行脚本：`./batch-summarize.sh`
3.  **结果**：你获得了一个全是结构化 Markdown 的新文件夹，可以直接导入 Wiki 系统。

#### 场景 B：基于知识的精准问答 (RAG)
日常工作中查询具体细节。

**操作步骤**：
```bash
# 询问跨文档的复杂问题
openclaw chat --agent domain-expert "根据后端规范和历史复盘报告，我们在处理高并发订单时有哪些禁止使用的数据库操作？请列出具体条款和来源。"
```
*OpenClaw 会自动从 `01_技术架构` 和 `04_历史项目` 两个目录中检索相关片段，综合后给出带出处的答案。*

### 第五阶段：进阶优化 (针对大量文档)

当文档数量达到数千级别时，需要注意以下优化：

#### 1. 增量更新策略
不要每次新增一个文件都 `--force` 全量重建。
*   OpenClaw 的 `autoIndex: true` 通常能监测文件变化。
*   手动触发增量索引：
    ```bash
    # 仅重新索引特定目录
    openclaw memory index --path ~/openclaw-knowledge-base/01_技术架构/新增规范
    ```

#### 2. 元数据过滤 (Metadata Filtering)
如果你的文档带有时间戳或版本号，可以在检索时过滤，避免查到旧规范。
*   在 `openclaw.json` 的 `retrieval` 配置中，可以添加 filter 逻辑（需配合支持 metadata 的向量后端，如 Chroma/Qdrant，若使用默认 SQLite 则主要靠文本内容区分）。
*   **Prompt 技巧**：在 System Prompt 中强调：“优先参考 2025 年以后的文档，若发现冲突，以最新版本为准。”

#### 3. 外部向量数据库 (可选)
如果本地 SQLite 性能瓶颈，可部署 **Qdrant** 或 **Chroma**，并在 `openclaw.json` 中切换 `memory-core` 的后端配置：
```json
"tools": [
  {
    "name": "memory-core",
    "config": {
      "vectorStore": {
        "type": "qdrant",
        "url": "http://localhost:6333",
        "collection": "company-knowledge"
      },
      ...
    }
  }
]
```

#### 4. 与 OpenCode 联动 (回顾之前的方案)
将整理好的结构化 Markdown (场景 A 的产出) 作为 OpenCode 的 `rules` 目录。
*   OpenClaw 负责“读万卷书”（处理原始 PDF）。
*   OpenCode 负责“行万里路”（读取提炼后的 Markdown 进行编码）。
*   **流程**：`PDFs` -> (OpenClaw Batch) -> `Structured MD` -> (OpenCode Context) -> `Code`.

### 常见问题排查 (FAQ)

| 问题现象 | 可能原因 | 解决方案 |
| :--- | :--- | :--- |
| **检索不到内容** | 切片过大或阈值过高 | 调小 `chunkSize` 至 512，降低 `threshold` 至 0.5；确认文件已被索引 (`openclaw memory status`)。 |
| **回答幻觉严重** | 检索片段不相关 | 开启 `hybridSearch: true`；在 Prompt 中加强“严禁编造”的指令；检查是否混入了低质量文档。 |
| **PDF 解析乱码** | 缺少解析器或字体 | 确保服务器安装了 `poppler-utils` (Linux) 或相应 PDF 解析依赖；尝试将 PDF 转为 TXT 后再摄入。 |
| **索引速度太慢** | 嵌入模型太慢 | 改用本地 Ollama 加速，或使用云端 Embedding API (如 Google/Voyage)；分批索引。 |

### 总结

这套流程将 OpenClaw 变成了一个**自动化知识工厂**：
1.  **配置**：定义好目录和 Agent 角色。
2.  **摄入**：一键向量化数百份文档。
3.  **提取**：批量生成结构化摘要（Markdown）。
4.  **应用**：支持精准 RAG 问答，并可无缝对接 OpenCode 辅助编程。

你可以先从**一个小分类（如 20 份文档）**开始跑通全流程，验证效果后再扩展到全量知识库。

## 代码检索

让 Agent **引用知识库里的代码**（而不仅仅是文档文本），是构建“代码辅助专家”的核心难点。因为代码具有**强逻辑性、依赖性和上下文关联性**，简单的向量检索往往只能找到片段，无法理解整体架构。

要在 OpenClaw 中实现高质量的代码引用，需要组合使用 **“精准检索策略”** + **“代码感知 Prompt"** + **“文件读取工具”**。

以下是完整实施方案：

---

### 核心挑战与解决方案

| 挑战 | 传统 RAG 的问题 | OpenClaw 解决方案 |
| :--- | :--- | :--- |
| **切片破碎** | 代码被切分成碎片，丢失函数/类结构。 | **智能分块**：按函数/类边界切片，保留完整逻辑单元。 |
| **依赖丢失** | 只检索到 `UserService`，没检索到它依赖的 `UserModel`。 | **多跳检索 (Multi-hop)** 或 **扩大上下文窗口**。 |
| **格式混乱** | 检索出的代码没有文件名、行号，无法定位。 | **强制元数据注入**：在 Prompt 中明确标注 `File: path`, `Lines: start-end`。 |
| **幻觉修改** | AI 随意修改引用的代码，导致错误。 | **只读模式约束**：在 System Prompt 中严禁修改原文，除非明确要求重构。 |

---

### 第一步：优化知识库摄入 (Ingestion)

代码文件的索引方式与普通文档不同。你需要确保 OpenClaw 在切片时**尊重代码结构**。

#### 1. 配置代码专用分块策略
在 `openclaw.json` 的 `memory-core` 配置中，针对代码文件调整参数：

```json
{
  "agents": {
    "definitions": {
      "code-expert": {
        "tools": [
          {
            "name": "memory-core",
            "config": {
              "indexing": {
                "watchPaths": ["/home/user/code-repo/src"],
                "fileExtensions": [".js", ".ts", ".py", ".go", ".java", ".md"],
                "chunkSize": 1500, 
                "chunkOverlap": 300,
                "codeAwareSplitting": true 
              },
              "retrieval": {
                "topK": 10, 
                "threshold": 0.6,
                "includeSources": true,
                "hybridSearch": true 
              }
            }
          },
          {
            "name": "fs-read", 
            "config": {
              "allowedPaths": ["/home/user/code-repo"]
            }
          }
        ]
      }
    }
  }
}
```
*注：`codeAwareSplitting` 是假设 OpenClaw 支持的特性（或通过 LangChain 的 `RecursiveCharacterTextSplitter` 配置语言特定分隔符）。如果版本不支持，需确保 `chunkSize` 足够大以容纳完整函数。*

#### 2. 关键：启用 `fs-read` 工具
仅靠向量检索往往不够。必须给 Agent 配备 **文件系统读取工具 (`fs-read`)**。
*   **作用**：当向量检索找到某个文件的关键片段时，Agent 可以主动调用 `fs-read` 读取**整个文件**或**相邻文件**，从而获得完整的上下文依赖。

---

### 第二步：编写“代码感知”System Prompt

这是最关键的一步。你需要告诉 Agent 如何正确处理检索到的代码。

在 `code-expert` 的 `systemPrompt` (或 `prompt.md`) 中加入以下规则：

```markdown
# Role
你是一名资深代码架构师，拥有项目完整源代码的知识库访问权限。

# 代码引用规范 (至关重要)
1. **精确溯源**：回答中涉及任何代码逻辑时，必须明确指出来源文件路径和大致行号。
   - ✅ 正确格式："在 `src/services/user.service.ts` (第 45-60 行) 中定义了..."
   - ❌ 错误格式："代码里有个地方写了..."

2. **完整性检查**：
   - 如果检索到的代码片段看起来不完整（如缺少 import 或函数结尾），**必须**调用 `fs-read` 工具读取完整文件后再回答。
   - 严禁基于碎片化信息猜测缺失的逻辑。

3. **依赖分析**：
   - 当用户询问某个函数时，主动思考它可能依赖的其他模块。如果需要，主动读取相关文件。

4. **修改建议**：
   - 如果用户要求修改代码，必须先展示**原始代码片段**（带引用），再展示**修改后的代码**，并解释差异。
   - 除非用户明确要求生成新文件，否则不要凭空创造不存在的文件路径。

5. **格式输出**：
   - 所有代码块必须使用 Markdown 代码块语法，并标注语言类型。
   - 长代码请折叠或只展示关键部分，避免刷屏。

# 工作流程
1. 接收用户问题。
2. 检索知识库获取相关代码片段。
3. **判断**：片段是否足以理解完整逻辑？
   - 否 -> 调用 `fs-read` 读取完整文件。
   - 是 -> 继续。
4. 结合上下文生成回答，严格标注来源。
```

### 第三步：实战工作流演示

#### 场景 A：查询现有代码逻辑
**用户提问**：“用户登录时的密码加密逻辑在哪里？用的是哪种算法？”

**Agent 内部执行过程**：
1.  **检索**：Vector DB 返回 `src/utils/crypto.js` 的片段，包含 `encryptPassword` 函数。
2.  **验证**：Agent 发现片段里没有 `import` 语句，不确定算法库。
3.  **工具调用**：自动执行 `fs-read(path="src/utils/crypto.js")` 读取全文件。
4.  **回答**：
    > 密码加密逻辑位于 `src/utils/crypto.js` (第 10-25 行)。
    > 使用的是 **bcrypt** 算法（见第 3 行 `const bcrypt = require('bcrypt');`）。
    >
    > **原始代码片段**：
    > ```javascript
    > // src/utils/crypto.js:10-25
    > async function encryptPassword(password) {
    >   const salt = await bcrypt.genSalt(10);
    >   return await bcrypt.hash(password, salt);
    > }
    > ```

#### 场景 B：基于现有代码进行扩展
**用户提问**：“参考 `OrderService` 的实现风格，帮我写一个 `RefundService`。”

**Agent 内部执行过程**：
1.  **检索**：找到 `src/services/order.service.ts`。
2.  **工具调用**：读取完整文件，分析其依赖注入方式、错误处理模式、日志记录风格。
3.  **生成**：模仿该风格生成新代码，并提示用户保存到新文件。

    > ```typescript

    > import { Transactional } from '@/decorators/transactional'; // 模仿 OrderService 的导入风格
    > ...
    > ```

### 第四步：进阶技巧 (提升准确率)

#### 1. 利用“代码地图” (Code Map)
如果项目很大，可以在知识库中维护一个手动编写的 `CODE_MAP.md` 文件，描述核心模块的文件路径和功能。
*   **内容示例**：
    ```markdown
    # 核心模块索引
    - 用户认证：`src/auth/` (包含 login.ts, jwt.strategy.ts)
    - 订单处理：`src/orders/` (包含 order.service.ts, payment.gateway.ts)
    ```
*   **作用**：Agent 可以先检索这个地图，快速定位文件路径，然后直接用 `fs-read` 读取，比纯向量检索更准。

#### 2. 限制检索范围 (Namespace Filtering)
如果用户明确问“前端代码”，可以在 Prompt 中引导 Agent 只在 `.ts/.tsx/.css` 文件中检索，避免混入后端逻辑干扰。
*   **技巧**：OpenClaw 的 `memory-core` 如果支持 metadata filter，可以配置 `filter: { extension: ".ts" }`。如果不支持，依靠 Prompt 约束：“优先检索前端相关文件”。

#### 3. 处理“找不到代码”的情况
在 Prompt 中明确兜底策略：
> “如果在知识库和文件系统中都找不到相关代码，请直接告知‘项目中未找到相关实现’，并建议用户检查路径或创建新文件。严禁编造不存在的文件路径。”

---

### 总结配置清单

要让 Agent 完美引用代码，请检查你的配置是否包含以下要素：

1.  **目录监控**：`watchPaths` 指向源码目录 (`src/`)。
2.  **双工具配置**：
    *   `memory-core`：用于模糊搜索定位文件。
    *   `fs-read`：用于读取完整文件以获取上下文。
3.  **System Prompt 约束**：
    *   强制要求标注 `File: Path` 和 `Lines`。
    *   强制要求在片段不全时调用 `fs-read`。
    *   禁止幻觉文件路径。
4.  **索引优化**：确保代码切片足够大，能容纳完整函数/类。

通过这套组合拳，你的 OpenClaw Agent 就不再是一个只会聊天的机器人，而是一个能**真正读懂、定位并引用你项目代码**的初级工程师。

## 检索需要的模块

实现**全文检索**、**自动引用关系提取**以及**导航自动更新**，是构建“活体”文献知识库的三大支柱。在 OpenClaw 中，这需要组合使用 **RAG 引擎**、**代码解释器 (Code Interpreter)** 和 **文件监听/自动化技能**。

以下是分模块的详细实施方案：

---

### 模块一：实现文献内容全文检索 (Full-Text Search)

OpenClaw 默认支持向量检索（语义搜索），但要实现精准的**全文检索**（关键词匹配、高亮显示），需要启用混合检索模式或集成全文索引引擎。

#### 方案 A：启用 OpenClaw 内置的混合检索 (推荐)
较新版本的 OpenClaw 内核通常支持 `hybrid_search` (向量 + BM25)。

1.  **配置知识库 (`openclaw.json`)**：
    ```json
    {
      "knowledge_base": {
        "name": "my-papers",
        "path": "~/papers/database",
        "embedding_model": "bge-large-zh-v1.5",
        "retrieval_config": {
          "mode": "hybrid", 
          "top_k": 5,
          "bm25_weight": 0.4, 
          "vector_weight": 0.6,
          "enable_rerank": true 
        }
      }
    }
    ```
    *   `mode: hybrid`: 同时使用关键词匹配 (BM25) 和语义向量。
    *   `enable_rerank`: 对初步召回的结果进行重排序，提高准确率。

2.  **使用方法**：
    直接在对话框输入关键词，系统会自动高亮匹配到的片段。
    ```bash
    openclaw chat "请检索所有包含 'LoRA' 且提到 '显存优化' 的文献段落，并列出来源文件名和页码。"
    ```

#### 方案 B：集成 Elasticsearch/Whoosh (企业级高性能)
如果需要毫秒级搜索海量文献：
1.  部署本地 Elasticsearch。
2.  编写 OpenClaw Skill `index-to-es`：每当有新 Markdown 文件生成，自动将其内容推送到 ES 索引。
3.  编写 Skill `search-es`：用户提问时，先查 ES 获取精准文档列表，再交给 LLM 总结。

---

### 模块二：文献间自动引用关系提取 (Citation Network)

这是最酷的功能：自动识别“论文 A 引用了论文 B”，并建立链接。这需要利用 **DOI 解析** 和 **正则匹配**。

#### 步骤 1：创建引用提取技能 (`skills/extract-citations/skill.md`)

这个技能的核心逻辑是：读取文献 -> 提取参考文献列表 -> 尝试匹配库内已有文献 -> 建立双向链接。

```markdown
# Skill: extract-citations
# 功能：扫描文献库，提取引用关系，构建知识图谱链接。

## 执行流程
1. **遍历**：扫描 `~/papers/database` 下所有 `.md` 文件。
2. **提取**：
   - 读取每个文件的 "References" 或 "Bibliography" 部分。
   - 使用正则提取 DOI、标题、作者。
   - 调用 `scholar-search` 工具补全缺失的 DOI。
3. **匹配 (核心)**：
   - 将提取出的参考文献 DOI/标题，与库中现有文件的 Frontmatter (`doi`, `title`) 进行比对。
   - **模糊匹配**：如果标题相似度 > 90%，视为同一篇。
4. **写入链接**：
   - 在当前的 Markdown 文件底部，添加一个 "Linked Papers" 章节。
   - 格式：`- [[被引论文文件名]] (引用自本文)`
   - **反向链接**：如果被引论文也在库中，自动在被引论文的底部添加 `- [[引用论文文件名]] (引用了本文)`。
5. **输出图谱数据**：生成一个 `citation_graph.json` 文件，格式为 `{"source": "A.md", "target": "B.md"}`，供可视化插件使用。

## 工具依赖
- `read_document`
- `scholar-search` (用于解析 DOI)
- `file_write`
```

#### 步骤 2：执行提取命令
```bash
# 全量构建引用关系
openclaw run-skill extract-citations --args dir=~/papers/database

# 或者针对单篇新论文
openclaw chat --agent paper-analyst "
读取 ~/incoming/new_paper.pdf，提取其参考文献列表。
检查这些参考文献是否已存在于我的知识库中。
如果存在，请在新生成的摘要文件底部添加 Markdown 内部链接 [[filename]]。
如果不存在，标记为 'External Citation'。
"
```

#### 步骤 3：可视化 (可选)
生成的 `citation_graph.json` 可以直接导入 **Obsidian Graph View** 或 **Gephi**，形成直观的文献引用网络图。在 Obsidian 中，只要使用了 `[[ ]]` 链接，图谱会自动生成。

---

### 模块三：添加文献后自动更新导航 (Auto-Update Navigation)

要实现“丢进文件 -> 自动处理 -> 自动更新目录”，需要建立一个**文件监听 (Watchdog)** 或 **Git Hook** 机制。

#### 方案 A：使用 OpenClaw 的 `watch` 模式 (最简单)
如果 OpenClaw 版本支持守护进程模式：

```bash
# 启动监听服务
openclaw watch --dir ~/incoming-papers --skill batch-paper-tagger --on-complete update-index
```
*逻辑：监测到 `~/incoming-papers` 有新文件 -> 触发 `batch-paper-tagger` 处理 -> 处理完成后触发 `update-index` 重建导航页。*

#### 方案 B：编写自动化 Shell 脚本 + Cron (通用性强)
创建一个脚本 `auto-ingest.sh`：

```bash
#!/bin/bash

INPUT_DIR=~/incoming-papers
DB_DIR=~/papers/database
LOG_FILE=~/papers/ingest.log

echo "[$(date)] 检查新文献..." >> $LOG_FILE

# 1. 移动新文件并触发处理
for file in $INPUT_DIR/*.pdf; do
  if [ -f "$file" ]; then
    filename=$(basename "$file")
    echo "[$(date)] 发现新文件：$filename" >> $LOG_FILE
    
    # 调用 OpenClaw 进行处理 (提取标签、摘要、分类)
    openclaw run-skill batch-paper-tagger \
      --args input_file="$file" \
      --args output_dir="$DB_DIR" >> $LOG_FILE 2>&1
    
    # 移动原文件到归档区 (可选)
    mv "$file" ~/papers/archive/
  fi
done

# 2. 提取引用关系 (增量更新)
# 只处理最近 1 小时生成的文件，避免全量扫描
recent_files=$(find $DB_DIR -name "*.md" -mmin -60)
if [ ! -z "$recent_files" ]; then
  echo "[$(date)] 更新引用关系..." >> $LOG_FILE
  openclaw run-skill extract-citations --args files="$recent_files" >> $LOG_FILE 2>&1
fi

# 3. 更新导航索引 (关键步骤)
echo "[$(date)] 重建导航索引..." >> $LOG_FILE
openclaw run-skill update-index \
  --args db_dir="$DB_DIR" \
  --args output_file="$DB_DIR/../index.md" >> $LOG_FILE 2>&1

# 4. (可选) 触发 MkDocs 重新构建
# mkdocs build
```

#### 方案 C：Git Hooks (适合版本控制用户)
如果你用 Git 管理文献库：
1.  在 `.git/hooks/post-commit` 或 `post-receive` 中添加上述脚本逻辑。
2.  每次 `git push` 新文献后，服务器/本地自动运行脚本，更新 `index.md` 和引用关系。
3.  CI/CD (如 GitHub Actions) 检测到 `index.md` 变化，自动重新部署 MkDocs 网站。

---

### 综合工作流演示

假设你下载了一篇新论文 `transformer_v2.pdf`：

1.  **放入**：你将文件拖入 `~/incoming-papers` 文件夹。
2.  **触发**：`auto-ingest.sh` (由 Cron 每分钟运行一次，或由 Watchdog 实时触发) 检测到文件。
3.  **处理 (Step 1)**：
    *   调用 `batch-paper-tagger`。
    *   AI 阅读 PDF，提取摘要、标签 (`[Transformer, Attention]`)，分类为 `基础架构`。
    *   生成 `~/papers/database/transformer_v2.md` (带 Frontmatter)。
4.  **关联 (Step 2)**：
    *   调用 `extract-citations`。
    *   AI 发现该文引用了库里的 `attention_is_all_you_need.md`。
    *   自动在 `transformer_v2.md` 底部写入 `[[attention_is_all_you_need]]`。
    *   自动在 `attention_is_all_you_need.md` 底部写入 `[[transformer_v2]] (引用了本文)`。
5.  **导航 (Step 3)**：
    *   调用 `update-index`。
    *   脚本重新扫描 `categories/` 目录。
    *   更新 `index.md` 中的表格，"基础架构" 类别数量 +1，并添加新链接。
    *   (若配置了 MkDocs) 自动触发 `mkdocs build`，网站实时更新。
6.  **完成**：你打开文献库网站或 Obsidian，发现新文章已经归类好、链接好，并且可以在搜索框里直接搜到了。

### 核心配置文件示例 (`skills/update-index/skill.md`)

为了让导航更新更智能，这个技能应该这样写：

```markdown
# Skill: update-index
# 目标：根据 database 目录下的最新文件，重构分类导航页。

## 逻辑
1. 读取 `~/papers/database` 下所有 `.md` 文件的 Frontmatter。
2. 按 `category` 分组统计数量。
3. 按 `date` 排序生成“最新入库”列表。
4. 按 `tags` 生成“热门标签云”。
5. 生成 `~/papers/index.md`，包含：
   - 统计概览卡片。
   - 分类导航表格 (带链接)。
   - 最新文献列表。
   - 标签云。
6. (可选) 生成 `categories/*.md` 子页面，每个页面列出该分类下的详细文献列表。
```

### 总结
实现这一整套自动化流程的关键在于：
1.  **标准化输入**：强制所有文献转为带 Frontmatter 的 Markdown。
2.  **模块化技能**：将“打标”、“摘引”、“建索引”拆分为独立技能。
3.  **自动化触发器**：利用 Shell 脚本、Cron 或 Watchdog 将这些技能串联成流水线。

一旦搭建完成，你的文献库将变成一个**自生长、自组织、自关联**的智能系统，你只需要负责“投喂”论文，剩下的全部交给 OpenClaw。

## 优先引用新文献

不能仅靠“口头嘱咐”，必须通过 **“检索层加权” + “提示词强约束” + “重排序机制”** 三位一体的策略来实现。

在 OpenClaw 架构下，以下是具体的配置方案：

---

### 策略一：检索层加权 (RAG Retrieval Weighting) —— 最根本
**原理**：在向量数据库检索阶段，直接给新文献更高的**分数权重**。这样即使旧文献语义匹配度稍高，新文献也会因为“时间加分”而排在前面，优先进入 AI 的上下文窗口。

#### 1. 配置 `openclaw.json` (混合检索 + 时间衰减)
启用支持元数据过滤和评分函数的向量数据库（如 Chroma, Qdrant, 或 OpenClaw 内置引擎）。

```json
{
  "retrieval": {
    "strategy": "hybrid_time_aware",
    "config": {
      "base_top_k": 10,
      "time_decay_factor": 0.1, 
      "recency_bonus": {
        "enabled": true,
        "rules": [
          { "condition": "year >= 2024", "score_boost": 0.5 },
          { "condition": "year >= 2022", "score_boost": 0.2 },
          { "condition": "year < 2020", "score_penalty": -0.3 }
        ]
      },
      "filters": {
        "must": [
          { "key": "status", "operator": "eq", "value": "active" }
        ]
      }
    }
  }
}
```
*   **效果**：一篇 2024 年的论文，即使语义相似度是 0.75，加上 0.5 的 bonus 后变成 1.25，会稳稳排在相似度 0.85 但年份为 2018 的论文之前。

#### 2. 动态时间窗口过滤
在用户提问涉及“最新进展”、“现状”、“趋势”等关键词时，自动收紧检索时间范围。

*   **OpenClaw Router 逻辑**：
    ```python
    if user_query contains ["最新", "进展", "现状", "2024", "2025", "recent"]:
        retrieval_config.filters.append({"key": "year", "operator": "gte", "value": 2023})
    ```

---

### 策略二：提示词强约束 (Prompt Engineering) —— 最直接
**原理**：在 System Prompt 中明确指令，要求 AI 对检索到的文档进行**二次筛选**，并强制规定引用优先级。

#### 1. 修改 Agent 的 System Prompt
在你的 `researcher` 或 `chat-agent` 配置中加入以下模块：

```markdown
# 📅 文献时效性优先协议 (Recency Priority Protocol)

你正在回答一个学术问题。请严格遵守以下引用规则：

1. **优先级排序**：
   - **第一梯队 (必须优先)**：近 2 年 (2024-2026) 的文献。只要存在相关新文献，**严禁**只引用旧文献。
   - **第二梯队 (补充参考)**：3-5 年前 (2021-2023) 的文献。仅当新文献不足时使用。
   - **第三梯队 (历史背景)**：5 年前 (<2021) 的文献。仅用于介绍基础理论或历史演变，**不得**作为当前技术状态的证据。

2. **冲突裁决**：
   - 如果新旧文献结论冲突，**必须以新文献为准**。
   - 话术示例：“虽然早期研究 (Author, 2018) 认为 X，但最新实证 (Author, 2025) 表明 Y 才是主流...”

3. **显式声明**：
   - 在回答开头，必须简要说明引用的时间范围。例如：“基于 2023-2026 年的最新研究...”
   - 如果被迫引用了超过 5 年的文献，必须加注："⚠️ 注：该观点源自较早研究，需结合最新进展审视。”

4. **无新文献处理**：
   - 如果检索结果中缺乏近 3 年文献，请直接告知用户：“目前库中关于此主题的最新研究仅更新至 XXXX 年，以下为基于现有资料的分析..." **切勿编造不存在的“最新”文献。**
```

#### 2. Few-Shot 示例 (增强理解)
在 Prompt 中提供正反例：

> **用户**：LLM 量化的最新方法有哪些？
>
> ❌ **错误回答**：Gong et al. (2018) 提出了量化感知训练... (忽略了 2024 年的新方法)
>
> ✅ **正确回答**：根据 **2024-2025 年** 的最新研究，主流方法已转向稀疏化量化。例如，Smith et al. (2025) 提出了... 而早期的 Gong et al. (2018) 方法虽奠定了基础，但在大模型场景下已不再适用。

---

### 策略三：重排序机制 (Re-Ranking) —— 最精准
**原理**：在检索出 Top-20 文档后，使用一个专门的 **Re-Ranker 模型** 或 **LLM 自身**，根据“相关性 + 时效性”重新排序，只把前 5 个（主要是新的）送给生成模型。

#### 1. 启用 Re-Ranker (如果 OpenClaw 支持)
配置 Cross-Encoder 模型，并在评分函数中加入时间因子。
```json
"reranker": {
  "model": "bge-reranker-v2-m3",
  "custom_scoring_formula": "0.7 * relevance_score + 0.3 * time_score"
}
```

#### 2. LLM 自我重排序 (Self-Re-Rank)
如果不方便部署重排模型，可以让 AI 在生成前先做一次“思维链”排序：

> **Prompt 指令**：
> “在生成最终回答前，请先执行以下步骤：
> 1. 列出检索到的所有候选文献及其年份。
> 2. 按年份从新到旧排序。
> 3. 剔除那些虽然相关但已被更新研究推翻的旧文献。
> 4. 仅选取排序前 5 名的文献作为核心依据生成回答。”

---

### 策略四：用户侧引导 (UI/UX)
在前端界面或默认提问模板中，帮用户加上时间约束。

*   **默认追问建议**：当用户问完一个问题，AI 回答后，底部按钮显示：
    *   “🔍 只看 2024 年以后的研究”
    *   “🔄 对比新旧观点差异”
*   **自动追加指令**：在发送请求给后端前，JS 自动在 query 后追加 `--min-year 2023` (如果系统支持这种语法糖)。

---

### 综合实战配置示例

将上述策略整合到 OpenClaw 的一个完整调用流程中：

**1. 用户提问**：
> "Transformer 架构有什么最新改进？"

**2. OpenClaw 处理流程**：
*   **Step 1 (路由识别)**：检测到“最新改进”，触发 `time_filter: year >= 2023`。
*   **Step 2 (检索)**：向量检索召回 20 篇文档。
*   **Step 3 (加权)**：应用 `recency_bonus`，2024-2026 的文档得分大幅提升，占据 Top 10 中的 8 席。
*   **Step 4 (重排序)**：Re-Ranker 再次确认，确保最相关的**新**文献排在最前。
*   **Step 5 (生成)**：LLM 收到包含 8 篇新文献的 Context，并根据 System Prompt 中的“优先协议”，生成如下回答：

> **AI 回答**：
> “基于 **2023-2026 年** 的最新文献，Transformer 架构的改进主要集中在效率优化和长上下文处理上：
> 1. **线性注意力机制**：最新的 **Zhang et al. (2025)** 提出了... (取代了早期的近似方法)。
> 2. **混合专家模型 (MoE)**：**Google (2024)** 的最新报告显示...
>
> *注：早期的 Vaswani et al. (2017) 奠定了基础，但上述新研究已解决了其计算复杂度高的问题。*"

### 总结
要让 AI 优先引用新文献，**不要只依赖 AI 的“自觉”**，必须：
1.  **物理上**：在检索阶段给新文献**加分**、给旧文献**设限**。
2.  **逻辑上**：在 Prompt 中规定**“有新必用新，无新要说明”**的硬性规则。
3.  **流程上**：引入**重排序**，确保新文献进入上下文窗口的核心位置。

这样配置后，你的 AI 就会像一个时刻关注前沿的科研助手，而不是一个抱着老黄历不放的图书管理员。


## 防止 AI 在对话中引用过时文献


核心在于建立一套从**数据源筛选**到**生成时约束**，再到**输出后验证**的全流程防线。

在 OpenClaw 架构下，你可以结合 **RAG 策略**、**Prompt 工程**和**元数据过滤**来实现这一目标。以下是具体实施方案：

---

### 第一道防线：数据源层面的“时间锁” (Data Source Filtering)
**原理**：在文献入库或检索阶段，直接物理隔离过时数据。这是最根本的解决方法。

#### 1. 入库时的“保质期”检查
在 `batch-paper-tagger` 技能中加入时间校验逻辑。如果文献发表年份早于设定阈值（如 5 年），自动标记为 `status: archived` 或 `warning: outdated`，并在 Frontmatter 中记录 `publish_year`。

```python
# 伪代码示例 (在 OpenClaw Skill 中)
def process_paper(file):
    meta = extract_metadata(file)
    current_year = 2026
    if current_year - meta['year'] > 5:
        meta['tags'].append('outdated-warning')
        meta['status'] = 'archived' # 默认不进入主检索库
    save_to_db(meta)
```

#### 2. 检索时的动态过滤 (Runtime Filtering)
配置 OpenClaw 的 RAG 检索器，使其在向量搜索时自动附加**元数据过滤器 (Metadata Filter)**。

**配置 `openclaw.json`：**
```json
{
  "retrieval_config": {
    "filters": {
      "must": [
        { "key": "status", "operator": "eq", "value": "active" },
        { "key": "publish_year", "operator": "gte", "value": 2021 } 
      ],
      "optional": [
        { "key": "tags", "operator": "contains", "value": "reviewed" }
      ]
    },
    "fallback_strategy": "warn_user" // 如果无新文献，提示用户而非强行引用旧的
  }
}
```
*效果：即使用户没提时间要求，AI 也只会从 2021 年以后的文献中检索内容。*

---

### 第二道防线：Prompt 层面的“强约束” (Prompt Engineering)
**原理**：通过 System Prompt 强制 AI 在生成回答时进行自我审查，优先使用新文献，并对旧文献保持警惕。

#### 1. 系统指令模板 (System Prompt)
在你的 `paper-analyst` 或 `chat-agent` 的 System Prompt 中加入以下模块：

```markdown
# ⚠️ 时效性严格约束 (Time-Sensitivity Protocol)

1. **优先原则**：在回答任何技术、数据或政策相关问题时，**必须优先引用近 3 年 (2023-2026)** 的文献。
2. **过时预警**：
   - 如果检索到的最佳匹配文献超过 5 年，必须在回答开头明确标注："⚠️ 注意：以下主要基于较早的研究 (年份)，该领域可能已有最新进展。”
   - 严禁将 5 年前的实验数据描述为“当前状态 (Current State)"。
3. **冲突处理**：如果新旧文献结论冲突，**以新文献为准**，并说明旧观点已被修正。
4. **无新文献处理**：如果近 3 年无相关文献，请直接回答：“经检索，近三年内未发现相关最新研究，以下为历史背景供参考..."，**不得编造新数据**。

# 引用规范
- 每条引用必须包含：[作者, 年份, 标题]。
- 若引用旧文献，需在括号内备注 "(历史参考)"。
```

#### 2. 用户侧的“时间锚点”技巧
教导用户在提问时带上时间约束（或者由前端 UI 自动追加）：
*   ❌ 错误问法：“LLM 量化的最新方法是什么？”
*   ✅ 正确问法：“请基于 **2024 年以后** 的文献，总结 LLM 量化的最新方法。忽略 2023 年以前的研究。”

### 第三道防线：生成后的“自检与验证” (Post-Generation Verification)
**原理**：让 AI 在输出答案前，先执行一次“事实核查”步骤，专门检查引用文献的年份。

#### 1. 引入“审查员 Agent” (Critic Agent)
在多 Agent 协作模式中，增加一个 `fact-checker` 角色。
*   **流程**：
    1.  `Researcher Agent` 生成草稿答案 + 引用列表。
    2.  `Fact-Checker Agent` 接收草稿，执行检查：
        *   “检查所有引用的 `publish_year`。”
        *   “是否有超过 5 年的文献被当作最新证据？”
        *   “是否有文献实际上已被后续研究推翻？”
    3.  如果有问题，退回重写；如果没问题，输出给用户。

#### 2. 自动化验证脚本
在 OpenClaw 的技能链中加入一步验证：
```python
def verify_citations(response):
    citations = extract_citations(response)
    for cite in citations:
        if cite.year < 2021 and "historical" not in response:
            return False, f"引用 {cite.title} ({cite.year}) 过旧且未标注历史背景。"
    return True, "Pass"
```

### 第四道防线：知识库的“动态维护机制” (Maintenance)
**原理**：文献库不是静态的，需要定期“除草”。

#### 1. 定期运行“过期扫描”技能
设置 Cron 任务，每月运行一次 `scan-outdated-papers`：
*   扫描库中所有文献。
*   检测是否有更新的综述论文发布（调用 `scholar-search`）。
*   如果发现某篇旧文献的核心观点已被新论文推翻，自动给旧文献打上 `deprecated` 标签，并在其摘要顶部添加警告：“⚠️ 此观点已被 [新论文标题] 修正。”

#### 2. 建立“黄金基准集” (Golden Set)
手动维护一个小规模的 `gold_standard.json`，包含该领域公认的、不可动摇的最新 SOTA 论文。
*   在 RAG 检索时，强制将这几篇论文作为**高权重上下文**注入，防止 AI 被大量低质量旧文献淹没。

### 总结：如何构建防过时体系？

| 层级 | 措施 | 关键点 |
| :--- | :--- | :--- |
| **数据层** | **元数据过滤** | 检索时强制 `year >= 202X`，旧数据打标归档。 |
| **指令层** | **Prompt 约束** | 明确要求“优先新文献”，“旧文献需标注”，“冲突时信新不信旧”。 |
| **流程层** | **Agent 自检** | 增加 `Fact-Checker` 角色，专门挑引用年份的刺。 |
| **维护层** | **定期清洗** | 自动扫描并标记被推翻的旧观点，保持库的“鲜活度”。 |

**💡 终极建议**：
不要试图完全禁止旧文献（因为经典理论永不过时）。正确的做法是**区分语境**：
*   问“最新进展” -> **只给新文献**。
*   问“发展历史” -> **可以给旧文献**。
*   问“基础原理” -> **新旧皆可，但需注明**。

通过 OpenClaw 的灵活配置，你可以让 AI 像一位严谨的学者一样，清楚地知道什么时候该看新论文，什么时候该翻老黄历。
