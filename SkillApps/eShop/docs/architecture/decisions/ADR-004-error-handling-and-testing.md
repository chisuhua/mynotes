# ADR-004: 错误处理与测试策略

**日期**: 2026-03-26  
**状态**: ✅ 已采纳  
**相关文档**: [[2026-03-26-ecommerce-analysis-system]]

---

## 背景

架构文档 v1.0.0 发布后的评审报告中，识别出两个 P0 优先级的改进项：

1. **错误处理策略缺失** — 文档未说明爬取失败、API 限流、Agent 超时等情况的处理策略
2. **测试策略缺失** — 文档未说明如何测试爬虫、Agent、Pipeline

这两个问题影响系统的可靠性和可维护性，需要在 v1.1.0 版本中解决。

## 决策

### 决策 1: 采用统一错误处理框架

**选项对比**：

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **方案 A: 渐进式** | 分层次逐步实现错误处理 | 实施成本低，可迭代 | 初期覆盖不全，需多次更新 |
| **方案 B: 统一框架** ⭐ | 设计统一框架，覆盖所有模块 | 一次设计全面覆盖，策略一致 | 初期设计复杂度高 |

**决策**：采用**方案 B：统一错误处理框架**

**理由**：
1. 避免多次文档更新，一次性完善
2. 错误处理策略一致，便于理解和维护
3. 直接回应评审报告 P0 建议

**框架核心**：
- **错误分类**：Transient、Permanent、Systematic、Timeout 四类
- **重试机制**：指数退避（1s, 2s, 4s + 随机抖动）
- **熔断器**：CLOSED → OPEN → HALF_OPEN 状态机
- **降级策略**：LLM→缓存→简化分析三级降级

---

### 决策 2: 采用测试金字塔模型

**测试分层**：
```
        ╱▔▔▔▔╲         E2E 测试（10%）
       ╱      ╲        集成测试（30%）
      ╱────────╲       单元测试（60%）
     ╱────────────╲    
    ╱______________╲   
```

**覆盖率要求**：
- **单元测试**：行覆盖率 ≥ 60%（核心模块 ≥ 80%）
- **集成测试**：关键路径 100%
- **E2E 测试**：主流程 100%

**决策理由**：
1. 测试金字塔是业界成熟模式
2. 60% 覆盖率是合理起点，不过度负担
3. 分层测试确保各层面质量

**详细文档**：测试策略已单独成文，详见 [TESTING-STRATEGY.md](../TESTING-STRATEGY.md)

## 权衡

### 正面后果 ✅

**错误处理框架**：
- 系统可靠性显著提升
- 故障恢复时间从分钟级降至秒级（重试）
- 熔断机制避免级联失败
- 错误日志便于问题排查

**测试策略**：
- 60% 覆盖率确保核心逻辑有测试保护
- 单元测试快速反馈（< 1 分钟）
- 集成测试验证 Agent 协作
- E2E 测试确保 Pipeline 可运行

### 负面后果 ⚠️

**错误处理框架**：
- 代码复杂度增加（熔断器状态管理）
- 需要额外学习成本（指数退避参数调优）
- 日志量增加（每次重试都记录）

**测试策略**：
- 测试代码量约为实现代码的 60%
- 需要维护测试 fixture 和 mock 数据
- CI/CD 运行时间增加（预计 +3-5 分钟）

## 实施计划

### 阶段 1: 错误处理核心（v1.1.0）

1. **实现重试工具类** — 指数退避装饰器
2. **实现熔断器** — 状态机 + 配置参数
3. **集成到爬虫模块** — 包装所有 HTTP 请求
4. **集成到 Agent 层** — LLM 调用保护
5. **添加错误日志** — 结构化日志格式

### 阶段 2: 测试框架（v1.1.0）

1. **搭建 pytest 框架** — 目录结构 + conftest
2. **编写单元测试** — 核心函数测试（覆盖率 60%）
3. **编写集成测试** — Agent 协作测试
4. **编写 E2E 测试** — Pipeline 脚本
5. **配置 CI/CD** — GitHub Actions 集成

### 阶段 3: 监控告警（v1.2.0）

1. **实现告警规则** — 失败率、熔断器、超时
2. **集成通知渠道** — 钉钉/企业微信
3. **添加仪表盘** — Grafana 可视化（可选）

## 技术细节

### 指数退避参数

```python
# 推荐参数
MAX_RETRIES = 3
BASE_DELAY = 1.0  # 秒
MAX_DELAY = 60.0  # 秒（上限）
JITTER = 1.0      # 随机抖动范围（秒）

# 延迟序列：1-2s, 2-3s, 4-5s
```

### 熔断器参数

```yaml
# 推荐配置
failure_threshold: 5        # 5 次失败触发熔断
recovery_timeout: 900       # 15 分钟后尝试恢复
half_open_max_calls: 1      # 试探模式只允许 1 个请求
success_threshold: 2        # 2 次成功恢复正常
```


## 迁移路径

### 现有代码改造

如已有代码无错误处理，按以下优先级改造：

1. **P0** — 爬虫 HTTP 请求（加重试）
2. **P0** — LLM API 调用（加熔断）
3. **P1** — 数据库操作（加超时）
4. **P2** — 文件 IO（加异常捕获）

### 测试覆盖扩展

1. **v1.1.0** — 核心模块 60% 覆盖率
2. **v1.2.0** — 核心模块 80% 覆盖率
3. **v1.3.0** — 整体 70% 覆盖率
