## 1. 双仓库架构回顾

|------|------|------|---------|
| **提案仓库** | `/workspace/mynotes/SkillApps/<project>/docs/architecture/` | 架构讨论、提案、评审 | `draft` → `under-review` → `approved` |

## 3. 状态追踪

### 3.1 提案状态机

```
draft → under-review → approved → synced
   ↑         ↓              ↓
   └───── rejected ←───────┘
```

| 状态 | 含义 | 允许操作 | 通知 |
|------|------|----------|------|
| `draft` | 草稿阶段 | 创建/修改/删除 | 无 |
| `under-review` | 评审中 | 修改（根据反馈） | @老板评审 |
| `approved` | 已批准 | 等待同步 | @DevMate 执行同步 |
| `synced` | 已同步 | 归档 | 无 |
| `rejected` | 已拒绝 | 归档，记录原因 | 无 |

### 3.2 状态文件

**位置**: `mynotes/.../architecture/proposals/YYYY-MM-DD-<主题>-status.md`

**模板**:
```markdown
# 架构提案状态

**提案**: YYYY-MM-DD-<主题>-proposal.md  
**创建时间**: YYYY-MM-DD HH:MM  
**创建人**: DevMate / 老板

## 状态流转

| 时间 | 状态 | 操作人 | 说明 |
|------|------|--------|------|
| 02:30 | draft | DevMate | 创建提案 |
| 02:35 | under-review | DevMate | 提交评审 |
| 02:40 | approved | 老板 | 批准同步 |
| 02:45 | synced | DevMate | 执行同步 |

## 评审意见

### DevMate 评审
- 需求完整性：8/10
- 架构一致性：✅ 符合现有设计
- 影响范围：模块 A, 模块 B

### 老板评审
- 批准理由：________
- 修改建议：________

## 同步信息

**同步时间**: YYYY-MM-DD HH:MM  
**同步文件**: 5 个  
**编码仓库版本**: commit abc123
```

## 4. 沟通规范

### 4.1 DevMate → 老板 通知

| 场景 | 通知方式 | 内容模板 |
|------|----------|----------|
| 提案创建 | 直接汇报 | "已创建架构提案：`<文件名>`，位置：`<路径>`，请评审" |
| 请求评审 | @老板 | "提案 `<名称>` 已更新，评审意见：`<摘要>`，请确认是否批准" |
| 请求同步 | @老板 | "架构已定稿，同步列表：`<文件数>` 个文件，请确认同步" |
| 同步完成 | 汇报 | "同步完成，报告：`SYNC-REPORT.md`，Git 提交：`<commit>`" |
| 发现架构问题 | @老板 | "编码发现架构问题：`<描述>`，建议：`<方案>`，请决策" |

### 4.2 老板 → DevMate 反馈

| 场景 | 反馈方式 | DevMate 行动 |
|------|----------|-------------|
| 批准提案 | "可以" / "同意" / "✅" | 进入阶段 2（定稿） |
| 需修改 | "需要修改：`<意见>`" | 返回 1.2（修改提案） |
| 拒绝 | "拒绝，原因：`<原因>`" | 归档，记录原因 |
| 批准同步 | "可以同步" / "同意同步" | 执行阶段 4（同步） |
| 暂缓 | "先不急" / "稍后再说" | 保持现状，记录原因 |

## 6. 检查清单

### 阶段 1：提案创建

- [ ] 需求完整性评分 ≥ 7/10
- [ ] 与现有架构一致性分析完成
- [ ] 至少 2 个方案对比
- [ ] 实施计划明确
- [ ] 风险评估完成

### 阶段 2：定稿

- [ ] 主文档更新（版本号 + 变更记录）
- [ ] ADR 创建/更新
- [ ] 同步列表生成
- [ ] 状态标记为 `approved`

### 阶段 3：同步确认

- [ ] 老板审查定稿文档
- [ ] 老板确认同步（口头/消息/标记）
- [ ] 记录确认时间

### 阶段 4：同步执行

- [ ] 执行同步脚本（--dry-run 预览）
- [ ] 执行同步脚本（实际同步）
- [ ] 生成 SYNC-REPORT.md
- [ ] Git 提交成功
- [ ] 通知编程助手（如有）

### 阶段 5：同步后验证

- [ ] DevMate 验证同步完整性
- [ ] 检查 SYNC-REPORT.md 内容
- [ ] Git 提交信息准确
- [ ] 老板确认（可选）

## 7. 常见问题

### Q1: 紧急架构变更如何处理？

**A**: 走快速通道：
1. DevMate 创建简要提案（1.2 简化版）
2. 老板快速评审（口头确认即可）
3. 同步执行
4. 事后补充完整文档

### Q2: 编码中发现严重架构问题怎么办？

**A**: 
1. 立即暂停编码
2. 写入 `temp/arch-issues.md`
3. @DevMate 评估
4. @老板决策（继续/修改/暂停）
5. 如需修改，返回阶段 1

### Q3: 老板长时间未评审怎么办？

**A**: 
- 超过 24 小时未评审 → DevMate 主动提醒
- 超过 48 小时 → 记录原因（老板忙/提案不清晰）
- 超过 1 周 → 归档提案，标记"过期"

### Q4: 同步失败如何处理？

**A**:
1. 检查同步脚本输出
2. 手动执行同步（cp 命令）
3. 记录失败原因到 `temp/sync-errors.md`
4. 修复后重新同步
